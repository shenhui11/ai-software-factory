# Architecture - FEAT-0001

## 选定方案概述

本功能采用人工已确认的 `solution_1 - 审稿驱动型章节工坊`，并严格纳入 `specs/decision/FEAT-0001.md` 中要求的增强项。系统定位为以章节续写为中心的 Web 创作平台，围绕“3 个剧情走向候选 -> 资深主编评分筛选 -> 写手生成正文 -> 资深读者评分 -> 低分回流 -> 保留最高分版本”形成稳定闭环。

架构目标有三点：
- 优先保证章节级可控性与可审计性，满足人工确认模式与全自动模式共存。
- 用最小可落地的结构化记忆维持跨章节连贯性，不引入未被决策选中的重型大纲中枢方案。
- 为模板、会员、运营后台与后续扩展预留清晰接口，但首期只实现 PRD 明确要求的范围。

## 总体架构

系统采用单体分层架构，前后端分离，后端以 FastAPI 提供业务 API 与任务编排入口。

核心组成如下：
- 创作前台：项目创建、章节工作台、方案选择与审稿结果展示、段落级重写/扩写、自动续写控制。
- 运营后台：模板管理、会员套餐与额度管理、订单管理、手动调额、内容安全策略、日志查询。
- 应用服务层：项目服务、章节服务、模板服务、会员服务、任务编排服务、内容安全服务。
- Agent 编排层：统一封装故事编辑 Agent、资深主编 Agent、写手 Agent、资深读者 Agent 的调用协议、提示词模板、超时与重试。
- 记忆与规则层：维护全书摘要、角色卡、世界观设定、已发生事件摘要、章节状态，并向 Agent 提供标准化上下文。
- 数据存储层：关系型数据库保存业务实体与生成记录；对象存储或文本字段保存正文、方案内容、差异版本与日志快照。

建议运行形态：
- Web 前端调用 FastAPI 同步接口完成项目管理、查询、人工确认、后台配置。
- 章节续写任务通过异步 Job Runner 串行执行章节流程，确保同一任务内章节顺序稳定，避免上下文错乱。
- 每章内部 revision 严格受状态机控制，最大回流 5 次。

## 模块划分

### 1. Project 模块
- 负责小说项目创建、基础设定维护、题材/篇幅/模板选择。
- 管理总纲、章节列表、项目模式配置（人工确认 / 全自动）。

### 2. Outline & Memory 模块
- 管理总纲、章节大纲、角色卡、世界观卡、事件摘要、章节状态。
- 为每章生成构建标准上下文包，供各 Agent 共用。
- 负责在章节完成后回写章节摘要与事件链。

### 3. Chapter Generation 模块
- 发起单次续写任务，校验最大连续章节数不超过 10。
- 驱动单章流程：走向生成、主编评分、正文生成、读者评分、回流重写、最终版本确认。
- 输出章节处理进度、评分明细、复核标记。

### 4. Agent Orchestration 模块
- 统一管理 4 类 Agent 的输入输出协议。
- 对每个 Agent 调用执行参数校验、日志记录、失败分类、可恢复重试。
- 保证每章固定先出 3 个走向方案，禁止跳过主编评分直接生成正文。

### 5. Rewrite & Expand 模块
- 支持对已生成正文的段落发起重写或扩写。
- 保存原文、修改稿、差异结果和一致性检查提示。

### 6. Membership & Quota 模块
- 管理免费章节额度、月度额度、套餐规则、订单、手动调额记录。
- 在任务发起前校验可用额度，在任务完成后按规则扣减。

### 7. Template 模块
- 支持系统模板与用户自定义模板。
- 管理模板草稿、审核、发布、标签与适用题材。

### 8. Admin & Compliance 模块
- 管理敏感内容策略、版权声明配置、日志留存。
- 为后台提供模板审核、会员管理、订单与安全策略查询能力。

## API 设计

以下为首期核心 API 形态，采用 REST 风格：

### 项目与章节
- `POST /api/projects`
  - 创建项目，输入题材、篇幅、模板 ID、总纲、角色设定、世界观设定。
- `GET /api/projects/{project_id}`
  - 查询项目详情、章节列表、当前模式、可用额度摘要。
- `PATCH /api/projects/{project_id}`
  - 更新项目基础信息与创作配置。
- `POST /api/projects/{project_id}/chapters/generate`
  - 发起续写任务，输入模式、起始章节、连续生成章数（1-10）。
- `GET /api/projects/{project_id}/tasks/{task_id}`
  - 查询任务状态、当前章节、评分结果、是否建议人工复核。
- `POST /api/projects/{project_id}/chapters/{chapter_id}/confirm`
  - 人工确认模式下采纳章节最终版本。

### 方案与正文
- `GET /api/chapters/{chapter_id}/outline-options`
  - 获取 3 个剧情走向方案及主编评分结果。
- `GET /api/chapters/{chapter_id}/drafts`
  - 获取正文版本列表、各 revision 分数、问题说明、最终选中状态。
- `POST /api/chapters/{chapter_id}/paragraph-rewrite`
  - 发起段落重写。
- `POST /api/chapters/{chapter_id}/paragraph-expand`
  - 发起段落扩写。

### 模板与会员
- `GET /api/templates`
  - 获取模板列表，区分系统模板与用户模板。
- `POST /api/templates`
  - 创建用户自定义模板。
- `PATCH /api/templates/{template_id}`
  - 编辑模板。
- `GET /api/membership/quotas`
  - 查询当前用户免费额度、月度额度、套餐信息。

### 后台
- `GET /admin/templates`
- `POST /admin/templates/{template_id}/publish`
- `GET /admin/memberships`
- `POST /admin/quotas/adjust`
- `GET /admin/orders`
- `GET /admin/safety/policies`
- `PATCH /admin/safety/policies/{policy_id}`
- `GET /admin/logs/tasks`

### 错误响应约定
- 统一返回结构：`code`、`message`、`details`、`request_id`
- 常见错误码：
  - `INVALID_ARGUMENT`
  - `QUOTA_EXCEEDED`
  - `TASK_CONFLICT`
  - `CONTENT_BLOCKED`
  - `AGENT_TIMEOUT`
  - `INTERNAL_ERROR`

## 状态流/权限流

### 章节状态流
1. `pending`
2. `outlining`：故事编辑 Agent 生成 3 个走向方案
3. `outline_selected`：资深主编 Agent 完成评分并选出最高分方案
4. `drafting`：写手 Agent 生成正文
5. `reviewing`：资深读者 Agent 评分
6. `revising`：低于 8.0 时回流重写，最多 5 次
7. `completed`：已有最终最高分版本
8. `needs_manual_review`：5 次回流后仍未达标，但输出最高分版本
9. `confirmed`：人工确认模式下用户最终采纳

### 任务状态流
- `queued` -> `running` -> `waiting_user_confirm` / `completed` / `failed`
- 人工确认模式下，单章完成后可暂停在 `waiting_user_confirm`
- 全自动模式下，在章节未超出本次 10 章上限前自动推进下一章

### 权限流
- 创作者用户：
  - 可管理自己的项目、模板、章节任务、段落改写结果
  - 不可访问后台管理接口
- 运营管理员：
  - 可管理模板审核、套餐、额度、订单、内容安全策略、任务日志
- 系统任务执行器：
  - 仅通过服务内部身份调用 Agent 编排与任务状态更新接口

## 模板/会员/AI协作（如适用）

### 模板
- 模板作为建项时的初始化配置，不承担独立编排中心职责。
- 首期模板字段以题材、世界观约束、角色模板、总纲框架、文风规则为主。
- 系统模板支持审核发布；用户模板默认私有。

### 会员
- 权益按章节生成次数、段落重写/扩写次数、月度额度管理。
- 单次续写最多 10 章属于产品固定上限，不因会员而突破本期规则。
- 套餐和手动调额变更必须可审计。

### AI 协作
- 故事编辑 Agent：固定输出 3 个走向方案，每个方案包含核心冲突、关键事件、结尾钩子。
- 资深主编 Agent：按情节推动、故事一致性、吸引力评分，自动选择最高分方案。
- 写手 Agent：基于选中走向和结构化记忆生成正文。
- 资深读者 Agent：按可读性、情绪张力、连贯性评分；低于 8.0 则回流。
- 每章最终保留所有 revision 中最高分版本；若未达标则附加“建议人工复核”标记。

## 错误处理

系统必须显式失败，不允许 silent fail。

关键策略如下：
- Agent 调用超时：标记当前 revision 失败，允许有限技术重试；超出重试上限则任务失败并写日志。
- 额度不足：在任务创建前拦截，返回 `QUOTA_EXCEEDED`。
- 内容安全命中：阻断输出入库，返回 `CONTENT_BLOCKED`，并记录命中策略。
- 数据冲突：同一项目存在运行中续写任务时，拒绝重复发起，返回 `TASK_CONFLICT`。
- 章节回流耗尽：不再继续重写，选择当前最高分版本落库并标记人工复核。
- 外部模型异常：记录 request_id、模型名、阶段、输入摘要，便于追踪。

## 可扩展性

首期架构预留以下扩展点，但不在本次实现范围内：
- 新增 Agent 角色时，只需在 Agent 编排层扩展协议与状态节点，无需改写项目主模型。
- 记忆层可从结构化摘要升级为更细粒度的事件图谱或检索式记忆。
- 任务执行器可从本地串行 Job Runner 平滑迁移到 Celery 等分布式队列。
- 模板可扩展版本化、推荐位、市场化分发。
- 后续配图、社区、数据看板、多格式导出可通过独立模块接入，不破坏当前章节主流程。

## 非功能约束

- 技术栈：后端必须使用 FastAPI；测试体系必须使用 pytest。
- 一致性：同一续写任务内章节串行执行，确保后章总是消费前章最终记忆结果。
- 性能：单次任务最多 10 章，单章最多 5 次回流，避免无限生成。
- 可观测性：核心流程必须记录任务日志、Agent 调用阶段日志、错误日志。
- 安全性：不得硬编码 secrets；后台管理接口需与普通创作接口隔离。
- 可靠性：所有状态流转必须可恢复、可审计，避免任务中断后丢失最高分版本。
- 可维护性：业务规则集中在服务层与状态机，不将阈值、回流上限散落在多个模块。
