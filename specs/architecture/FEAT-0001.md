# Architecture - FEAT-0001

## 选定方案概述
本项目首期采用 `solution_1 - 审稿驱动型章节工坊`，并严格执行人工决策中的补充要求。系统以“章节”为最小交付单元，围绕单章生成建立可审、可回流、可自动推进的多 Agent 协作流水线。

首期固定能力如下：
- 支持人工确认模式与全自动模式两种创作模式。
- 每章先由故事编辑 Agent 生成 3 个剧情走向方案。
- 资深主编 Agent 对 3 个方案按“情节推动、故事一致性、吸引力”评分，并自动选择最高分方案。
- 写手 Agent 基于入选方案生成正文。
- 资深读者 Agent 对正文按“可读性、情绪张力、连贯性”评分，低于阈值时最多回流 5 次。
- 每章保留评分最高的正文版本；若 5 次后仍未达阈值，则输出最高分版本并标记“建议人工复核”。
- 单次连续续写最多 10 章，章节间严格串行执行。
- 连贯性记忆采用结构化记忆，至少包含全书摘要、角色卡、世界观设定、已发生事件摘要、章节状态。
- 段落级重写/扩写必须包含上下文一致性检查与差异对比。

本架构不恢复任何未被人工确认的额外能力，尤其不引入多人实时协作、社区能力、配图、多格式导出、复杂企业审批流等范围外内容。

## 总体架构
系统采用前后端分离的 Web 架构，后端以 FastAPI 为唯一服务框架，围绕“同步业务 API + 异步任务编排 + 结构化记忆存储”组织。

核心分层如下：
- 前端工作台：项目创建、章节工作台、大纲编辑器、候选方案面板、审稿台、差异对比视图、自动任务面板、后台运营页。
- FastAPI 应用层：提供项目、章节、模板、会员、运营、Agent 任务查询与控制 API。
- 编排层：负责章节生成状态机、Agent 调用顺序、回流控制、10 章串行续写控制、任务日志记录。
- 领域服务层：封装项目管理、模板应用、记忆组装、章节评分、版本选择、额度扣减、内容安全校验。
- 持久化层：关系型数据库存储业务实体；对象存储或文本字段存储长文本正文；审计日志独立留存。
- AI 适配层：统一封装故事编辑、资深主编、写手、资深读者 4 类 Agent 调用接口，屏蔽具体模型差异。

建议部署形态：
- `apps/api`: FastAPI HTTP 服务。
- `apps/worker`: 异步任务 Worker，执行章节流水线与回流逻辑。
- `packages/domain`: 领域模型、服务、状态机、权限与配额规则。
- `packages/integrations`: LLM/存储/消息队列/审计日志等基础设施适配。

数据流主线：
1. 用户通过前端创建项目、维护总纲与结构化设定。
2. 前端调用 FastAPI 创建章节生成任务。
3. 编排层读取项目上下文与结构化记忆，按章节顺序触发多 Agent 流程。
4. 每章完成后写入候选方案、主编评分、正文版本、读者评分、最终版本和章节状态。
5. 章节完成后更新结构化记忆，再进入下一章；若达到 10 章上限或用户停止则任务结束。

## 模块划分
### 1. 项目与大纲模块
- 管理小说项目基础信息：题材、篇幅、模板、创作模式。
- 管理总纲、章节目标、角色设定、世界观设定。
- 提供章节列表与章节状态视图。

### 2. 模板模块
- 支持官方模板与用户私有模板。
- 模板结构至少包含题材约束、世界观/角色预设、章节规划框架、风格约束。
- 后台支持模板审核、发布、下架和版本管理。

### 3. 章节编排模块
- 负责单章与连续续写任务的状态机。
- 控制“3 个方案 -> 主编评分选案 -> 正文生成 -> 读者评分 -> 回流”的固定顺序。
- 强制单次任务最多 10 章，且章节间串行。

### 4. Agent 协作模块
- 故事编辑 Agent：产出 3 个剧情走向方案。
- 资深主编 Agent：对方案评分并选择最高分。
- 写手 Agent：生成章节正文。
- 资深读者 Agent：评分并输出问题说明，驱动回流。

### 5. 结构化记忆模块
- 维护全书摘要、角色卡、世界观设定、已发生事件摘要、章节状态。
- 为章节生成和段落级改写提供统一上下文拼装。
- 在章节确认或自动完成后更新记忆快照。

### 6. 段落改写模块
- 支持对指定段落发起重写或扩写。
- 调用一致性检查器比较改写前后与全章上下文是否冲突。
- 输出差异对比结果与一致性提示。

### 7. 会员与配额模块
- 管理免费章节额度、月度额度、套餐权益、订单与手动调额。
- 在章节生成、回流、局部改写等操作前执行配额校验与扣减。

### 8. 内容安全与审计模块
- 执行敏感题材过滤、版权声明展示、数据留存规则。
- 对核心生成链路、评分结果、人工确认动作、后台调额动作写审计日志。

## API 设计
以下为首期建议 API 结构，均由 FastAPI 提供，响应统一包含 `code`、`message`、`data`，错误场景禁止 silent fail。

### 项目与大纲
- `POST /api/v1/projects`
  - 创建项目，输入题材、篇幅、模板、创作模式。
- `GET /api/v1/projects/{project_id}`
  - 获取项目详情与聚合状态。
- `PATCH /api/v1/projects/{project_id}`
  - 更新项目基础信息。
- `PUT /api/v1/projects/{project_id}/outline`
  - 保存总纲、角色设定、世界观设定。
- `PUT /api/v1/projects/{project_id}/chapters/{chapter_id}/brief`
  - 保存章节目标与章节大纲。

### 章节生成与审稿
- `POST /api/v1/projects/{project_id}/chapters/{chapter_id}/generate`
  - 发起单章生成任务，可指定人工确认或自动执行。
- `POST /api/v1/projects/{project_id}/autopilot-runs`
  - 发起连续续写任务，`target_chapter_count <= 10`。
- `GET /api/v1/generation-runs/{run_id}`
  - 查询任务状态、当前章节、当前回流次数、已完成章节列表。
- `POST /api/v1/chapters/{chapter_id}/outline-options/{option_id}/confirm`
  - 人工确认模式下选择剧情方案。
- `POST /api/v1/chapters/{chapter_id}/finalize`
  - 人工确认最终章节版本入库。

### 段落级重写/扩写
- `POST /api/v1/chapters/{chapter_id}/paragraph-edits`
  - 输入段落范围、操作类型（rewrite/expand）、补充指令。
  - 返回新版本文本、差异对比、一致性检查结果。

### 模板与后台运营
- `GET /api/v1/templates`
- `POST /api/v1/templates`
- `PATCH /api/v1/templates/{template_id}`
- `POST /api/v1/admin/templates/{template_id}/review`
- `GET /api/v1/admin/membership-plans`
- `POST /api/v1/admin/membership-plans`
- `POST /api/v1/admin/quotas/adjustments`
- `GET /api/v1/admin/orders`

### 关键错误码建议
- `400` 参数非法或超过约束，例如连续续写章节数大于 10。
- `403` 无权限访问项目或后台资源。
- `409` 章节状态冲突，例如未完成上一章却请求后续章自动推进。
- `422` 上下文缺失，例如项目未配置总纲即发起生成。
- `429` 配额不足或触发会员限制。
- `503` Agent 调用失败或异步任务系统不可用。

## 状态流/权限流
### 章节状态流
建议章节状态：
- `draft_brief`
- `outline_generating`
- `outline_ready`
- `outline_confirmed`
- `draft_generating`
- `reviewing`
- `revision_pending`
- `completed`
- `completed_with_manual_review_flag`
- `failed`

状态转换规则：
- 人工确认模式下，`outline_ready -> outline_confirmed` 需要用户显式确认。
- 全自动模式下，主编评分完成后直接写入 `outline_confirmed`。
- 读者评分 `< 8.0` 时进入 `revision_pending`，最多重复 5 次。
- 达到通过阈值后进入 `completed`。
- 超过 5 次仍未达标时，保留最高分版本并进入 `completed_with_manual_review_flag`。

### 连续续写状态流
- 创建任务时校验项目权限、配额、目标章节数是否小于等于 10。
- Worker 串行执行章节任务；仅当前章完成并完成记忆更新后，下一章才可启动。
- 任一章节硬失败时，任务进入 `partial_failed` 或 `failed`，并保留已完成章节结果。

### 权限流
- 创作者用户：管理自己的项目、模板、章节、局部改写任务。
- 后台运营用户：管理模板审核、会员套餐、订单、调额、内容安全策略。
- 首期不实现复杂 RBAC，采用“前台用户 / 后台运营”两级权限模型即可。
- 所有写操作均需校验资源归属；后台接口需额外校验管理员身份。

## 模板/会员/AI协作（如适用）
### 模板
- 模板是建项初始化器，不直接替代项目自身结构化记忆。
- 系统模板可预置题材框架、角色模板、世界观约束、章节节奏建议。
- 用户模板默认为私有；后台审核发布仅针对系统模板或拟公开模板。

### 会员
- 会员权益与实际生成链路绑定，不做脱离业务的抽象计费。
- 首期建议将配额计量聚焦到：
  - 章节生成次数
  - 段落重写/扩写次数
  - 自动续写任务可用次数
  - 单次自动续写最大章节数上限（封顶仍不超过 10）
- 回流产生的额外生成消耗需在业务规则中明确，可配置为计入或不计入套餐消耗，但实现层必须支持可配置。

### AI 协作
- 固定为四段串行协作，不引入并行章节生成，避免上下文错位。
- Agent 统一通过 `AgentRunner` 抽象调用，输入输出强制结构化：
  - 故事编辑输出 3 个 `OutlineOption`
  - 主编输出评分明细和 `selected_option_id`
  - 写手输出 `ChapterDraft`
  - 读者输出评分明细、问题列表、是否回流建议
- 所有 Agent 输出均需做 schema 校验，避免不规范结果污染状态机。

## 错误处理
系统必须显式处理以下失败场景：
- 参数错误：返回明确字段级错误，阻止非法任务进入编排层。
- 额度不足：在任务启动前阻断，并返回当前套餐与剩余额度。
- 状态冲突：例如章节未完成却尝试跳章自动续写，返回 `409`。
- Agent 输出不合法：记录原始响应和校验错误，章节置为失败或待人工处理。
- 单次调用失败：支持有限次基础重试；超过阈值后标记任务失败，不得 silent fail。
- 回流上限达到但未达标：正常产出最高分版本，并显式打上“建议人工复核”标记。
- 内容安全命中：阻断发布或继续生成，并记录命中规则。

日志与审计要求：
- 记录任务开始、Agent 调用、评分结果、回流次数、最终选中版本、人工确认动作、后台调额动作。
- 日志需关联 `project_id`、`chapter_id`、`generation_run_id` 便于追踪。

## 可扩展性
首期架构在不扩大 MVP 范围的前提下预留以下扩展点：
- Agent 扩展：可新增插画 Agent、设定校对 Agent、出版格式化 Agent，但不影响现有四段流水线接口。
- 记忆扩展：可从快照模型演进到版本化事件流或更细粒度知识卡。
- 任务扩展：可从单 Worker 串行执行演进为“项目内串行、项目间并行”的队列模型。
- 数据扩展：可为后续社区、数据看板、多格式导出增加读取模型，而不改写创作主库。
- 权限扩展：两级权限模型可平滑升级到更细粒度 RBAC。

## 非功能约束
- 技术栈：后端必须使用 FastAPI；测试必须使用 pytest。
- 一致性优先：章节间必须串行，单次自动续写最多 10 章，单章最多回流 5 次。
- 可观测性：核心流程必须有结构化日志，禁止 silent fail。
- 性能：单章任务应支持异步执行，避免阻塞前端请求；前端通过轮询或事件流查看进度。
- 可靠性：任务状态更新与版本落库应具备事务边界，避免“正文已生成但状态未更新”。
- 安全性：不得硬编码 secrets；后台接口必须校验管理员身份；敏感内容策略必须可配置。
- 合规性：需支持版权声明展示、必要的数据留存与后台操作审计。
- 变更约束：首期以最小可交付架构为准，不引入 decision 未确认的新能力，不恢复任何未批准范围。
