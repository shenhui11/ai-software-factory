# FEAT-0001 Architecture

## 1. 模块边界

### 1.1 总体架构

`FEAT-0001` 采用单体式 FastAPI 后端 + 单人 Web 工作台的 MVP 架构。首版优先保证编辑闭环可用性，不引入分布式服务、复杂工作流引擎或多人协作子系统。

推荐分层如下：

- Presentation Layer：FastAPI 路由层，负责请求校验、响应序列化、错误映射。
- Application Layer：用例编排层，负责故事创建、骨架生成、章节生成、段落编辑、QA 扫描、定点修复、版本回退、导出等业务流程。
- Domain Layer：核心领域模型与规则，负责 Story Project、Chapter、Version、QA Issue、Character Profile 等实体与约束。
- Infrastructure Layer：持久化、LLM 网关、日志、配置、导出器。

### 1.2 模块拆分

建议按能力域拆分为以下模块：

- `project`：故事项目管理，维护题材、风格、目标读者、篇幅目标、故事基调等基础元数据。
- `canon`：核心设定面板，维护角色卡、世界观摘要、风格约束，作为生成、编辑、QA 的统一上下文来源。
- `outline`：故事骨架生成，输出一句话梗概、核心冲突、主要角色摘要、章节级简纲。
- `chapter`：章节草稿管理，负责章节创建、正文保存、当前工作版本加载。
- `edit`：段落级编辑能力，支持重写、扩写、压缩、语气调整、叙述风格调整。
- `qa`：章节 QA 扫描，输出结构化问题列表和修复建议。
- `revision`：基于单条 QA 问题的局部修复，限定修改范围，不允许默认整章重写。
- `versioning`：章节级与段落级版本快照、候选版本保留、diff 对比、回退。
- `export`：导出为 Markdown 或纯文本。
- `platform`：统一配置、日志、错误结构、健康检查、LLM 提供方适配。

### 1.3 关键边界约束

- `outline` 只负责生成结构化骨架，不直接写入章节正文。
- `edit` 只处理用户指定文本片段，不直接触发全章重建。
- `qa` 只负责检测与诊断，不直接修改内容。
- `revision` 必须依赖 `qa` 输出的问题定位信息执行定点修复。
- `versioning` 负责所有可回退操作，其他模块不得直接覆盖历史版本。
- `canon` 是唯一的“长期设定上下文”入口，避免角色卡、风格约束散落在多个模块中。

## 2. API 设计

### 2.1 API 约定

- 风格：REST JSON API。
- 路径前缀：`/api/v1`。
- 错误结构统一为：

```json
{
  "error": {
    "code": "string_code",
    "message": "human_readable_message",
    "details": {}
  },
  "request_id": "uuid"
}
```

- 长耗时生成/扫描操作在 MVP 可先同步返回；若后续响应时间不可接受，再演进为异步任务模型。

### 2.2 核心接口

#### 项目与设定

- `POST /api/v1/projects`
  - 创建故事项目。
  - 请求体：`title`、`genre`、`style`、`target_audience`、`length_target`、`tone`、`premise`。
- `GET /api/v1/projects/{project_id}`
  - 获取项目详情与当前状态。
- `PATCH /api/v1/projects/{project_id}`
  - 更新项目基础信息。
- `GET /api/v1/projects/{project_id}/canon`
  - 获取核心设定面板。
- `PUT /api/v1/projects/{project_id}/canon`
  - 整体更新角色卡、世界观摘要、风格约束。

#### 骨架与章节

- `POST /api/v1/projects/{project_id}/outline:generate`
  - 基于项目参数和核心设定生成故事骨架。
  - 返回：一句话梗概、核心冲突、角色摘要、章节级简纲数组。
- `GET /api/v1/projects/{project_id}/outline`
  - 读取当前骨架。
- `POST /api/v1/projects/{project_id}/chapters`
  - 创建章节元数据，可基于简纲项初始化。
- `POST /api/v1/chapters/{chapter_id}/draft:generate`
  - 生成章节初稿。
  - 输入：章节目标、关联简纲、上下文窗口策略。
- `GET /api/v1/chapters/{chapter_id}`
  - 获取章节详情、当前正文、当前活动版本。
- `PATCH /api/v1/chapters/{chapter_id}`
  - 保存人工编辑后的章节内容。

#### 段落编辑与 QA

- `POST /api/v1/chapters/{chapter_id}/edits`
  - 对选中文本执行编辑。
  - 输入：`selection_start`、`selection_end`、`operation`、`instruction`。
  - `operation` 限定为：`rewrite`、`expand`、`compress`、`tone_shift`、`style_shift`。
  - 返回：一个或多个候选改写结果，不直接覆盖正文。
- `POST /api/v1/chapters/{chapter_id}/qa-scans`
  - 发起章节 QA 扫描。
  - 返回结构化问题列表，类型至少覆盖：逻辑跳跃、设定冲突、情节断裂、信息重复、伏笔未承接、角色口吻偏移。
- `GET /api/v1/chapters/{chapter_id}/qa-scans/{scan_id}`
  - 获取某次 QA 扫描结果。

#### 定点修复、版本、导出

- `POST /api/v1/qa-issues/{issue_id}/fix`
  - 基于单条 QA 问题进行局部修复。
  - 输入：修复策略、允许修改范围。
  - 返回：修复候选版本与影响片段范围。
- `GET /api/v1/chapters/{chapter_id}/versions`
  - 获取章节版本列表。
- `POST /api/v1/chapters/{chapter_id}/versions`
  - 创建章节快照版本。
- `GET /api/v1/versions/{version_id}/diff?base_version_id=...`
  - 获取文本差异。
- `POST /api/v1/versions/{version_id}:restore`
  - 回退到指定版本。
- `GET /api/v1/projects/{project_id}/export?format=md|txt`
  - 导出整部作品或已完成内容。

### 2.3 内部服务接口

应用层内部建议定义以下服务接口，避免路由直接拼接提示词或操作存储：

- `OutlineGenerationService`
- `ChapterDraftService`
- `ParagraphEditService`
- `ChapterQaService`
- `IssueFixService`
- `VersioningService`
- `ExportService`
- `LlmGateway`

### 3.2 存储建议

- 关系型数据库：存储项目、章节、版本元数据、QA 结果、角色卡。
- 大文本字段：章节正文、版本正文、导出快照可先放在数据库文本列；当文本体量上升后再迁移对象存储。
- 不建议首版引入向量库；如需增强长上下文检索，可后续在 `canon` 与历史章节上增量演进。

### 3.3 一致性约束

- 一个 `Chapter` 只能有一个 `current_version_id`。
- 任意自动生成或修复结果都必须先形成 `ChapterVersion` 或候选结果，再由用户确认采用。
- `QaIssue` 必须绑定到具体扫描结果与正文偏移范围，确保“问题到修复”的链路可追踪。

## 4. 权限与安全

### 4.1 权限模型

首版产品明确为单人 Web 写作工作台，不做多人协作与细粒度角色权限。MVP 权限模型保持最小化：

- 单用户拥有其项目下全部读写权限。
- 所有写操作默认作用于当前用户自己的故事项目。
- 架构上在 API 层保留 `owner_id` 或等价字段扩展位，为后续接入认证/授权做准备。

### 4.2 安全要求

- 不允许硬编码 LLM API Key、数据库密码或其他敏感配置，统一从环境变量或配置中心读取。
- 所有写接口进行输入校验，限制最大文本长度、章节长度、角色卡数量，防止超大请求拖垮服务。
- 所有编辑、修复接口必须显式提供修改范围，避免因提示词失控导致全章误改。
- 导出接口只允许导出当前用户可访问项目的数据。
- 日志中禁止记录完整密钥、令牌和敏感配置；正文内容只记录摘要或长度，不记录全量文本。

### 4.3 失败保护

- 生成、编辑、修复流程在写入正式版本前必须先落候选结果或版本快照。
- 当 LLM 返回异常、超时或结果不可解析时，接口返回统一错误结构，不允许 silent fail。
- 回退操作必须基于已存在版本，不允许覆盖式恢复且不保留历史。

## 5. 可观测性

### 5.1 日志

核心流程必须有结构化日志，至少覆盖：

- 项目创建
- 骨架生成
- 章节初稿生成
- 段落编辑
- QA 扫描
- QA 定点修复
- 版本创建与回退
- 导出

每条日志应带：

- `request_id`
- `project_id`
- `chapter_id`（如适用）
- `operation`
- `status`
- `latency_ms`
- `llm_provider`（如适用）

### 5.2 指标

建议暴露以下基础指标：

- HTTP 请求总量、状态码分布、接口耗时
- LLM 调用次数、成功率、超时率、平均耗时
- 骨架生成成功率
- 章节初稿生成成功率
- QA 扫描平均问题数
- 局部修复采纳率
- 版本回退次数

### 5.3 追踪与诊断

- 每次生成、编辑、QA、修复都生成唯一操作 ID，串联 API 日志、LLM 调用日志与版本记录。
- QA 问题与修复结果之间保留 `issue_id -> version_id` 关联，便于分析“哪些问题被修复、修复后是否被采纳”。
- 对 LLM 原始响应仅保留必要调试片段或摘要，避免成本和隐私风险。

## 6. 部署说明

### 6.1 运行形态

首版建议部署为单个 FastAPI 应用进程，对外提供 REST API；前端工作台通过 HTTP 调用。

### 6.2 依赖组件

- FastAPI 应用
- 关系型数据库（建议 PostgreSQL，MVP 也可先用 SQLite 做本地开发）
- LLM 提供方 API
- 可选：对象存储，用于后续大型导出文件或版本快照归档

### 6.3 配置项

至少包含以下环境变量：

- `APP_ENV`
- `DATABASE_URL`
- `LLM_PROVIDER`
- `LLM_API_BASE`
- `LLM_API_KEY`
- `LOG_LEVEL`
- `MAX_CHAPTER_LENGTH`
- `MAX_EDIT_SELECTION_LENGTH`

### 6.4 部署建议

- 开发环境：`uvicorn` 启动 FastAPI，使用本地数据库。
- 生产环境：使用 `gunicorn + uvicorn worker` 或等价 ASGI 部署方案。
- 数据库迁移必须独立管理，不能靠应用启动时隐式修改表结构。
- 如果后续将生成、QA 扫描改为异步任务，可引入任务队列，但不作为首版前置依赖。

## 7. 风险与技术债

### 7.1 主要风险

- 骨架质量不稳定：若故事骨架偏弱，后续编辑链路价值会被削弱。
- 局部编辑边界失控：段落改写或 QA 修复若扩大影响范围，会直接伤害用户对“可控性”的信任。
- QA 误报与泛化：若问题提示过于空泛，用户会认为诊断不可用。
- 长上下文一致性不足：多章节场景下，角色口吻、设定和伏笔容易漂移。
- 版本爆炸：频繁编辑会快速累积版本，若没有归档与筛选策略，后续读取和比较成本会上升。

### 7.2 首版接受的技术债

- 同步生成接口：首版可接受长请求阻塞，但后续需要视耗时改造为异步任务。
- 无向量检索：首版主要依赖核心设定面板和有限上下文拼接，长篇一致性能力有限。
- 文本 diff 基础化：MVP 可先用纯文本 diff，不做复杂可视化差异渲染。
- 单体部署：首版不拆分生成服务、QA 服务、导出服务，以降低交付复杂度。

### 7.3 后续演进方向

- 引入异步任务与任务状态查询，优化长耗时生成和扫描体验。
- 为 `canon` 与历史章节增加检索增强能力，提升跨章节一致性。
- 为 QA 问题增加更稳定的严重程度判定与修复策略模板。
- 在不破坏单人模型前提下，为未来多人协作预留认证、所有权和审计字段。
