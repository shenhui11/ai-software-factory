# Architecture - FEAT-0001

## 选定方案概述
本功能最终采用 `solution_1 - 审稿驱动型章节工坊`，并严格吸收 `specs/decision/FEAT-0001.md` 中的人工作业修改，不回退到任何未被选定的替代方案。

首期能力边界固定如下：
- 同时支持人工确认模式与全自动续写模式。
- 全自动模式单次任务最多连续生成 10 章。
- 每章必须先生成 3 个剧情走向方案。
- 必须新增并使用“资深主编 Agent”对 3 个走向按情节推动、故事一致性、吸引力评分，并自动选择最高分方案。
- 写手 Agent 基于被选中走向生成章节正文。
- 资深读者 Agent 对正文按可读性、情绪张力、连贯性评分，分数低于 `8.0` 时自动回流给写手 Agent 重写，单章最多回流 5 次。
- 若 5 次回流后仍未达标，系统仍输出该章评分最高版本，并标记“建议人工复核”。
- 首期必须具备结构化记忆能力，至少覆盖全书摘要、角色卡、世界观设定、已发生事件摘要、章节状态。
- 首期必须提供基础任务状态展示，能查看章节进度、评分、回流次数和最终结果。

明确不纳入本架构的能力：
- 多用户实时协作
- 社区发布与互动
- AI 配图
- 多格式导出
- 复杂企业级审批流
- 超出基础运营范围的高级 BI 与精细化成本治理

## 总体架构
系统采用前后端分离的 Web 架构，后端统一使用 FastAPI，围绕“同步管理 API + 异步章节流水线 + 结构化记忆存储 + 运营后台”组织。

建议逻辑分层如下：

1. 表现层
- 创作前台：项目创建页、小说工作台、大纲编辑器、候选方案面板、正文审稿区、自动续写任务面板、段落差异对比区。
- 运营后台：模板审核、会员套餐、额度调整、订单管理、内容安全策略、日志查看。

2. API 层
- 由 FastAPI 提供 REST API。
- 负责鉴权、参数校验、响应封装、错误码映射。
- 对外暴露项目、章节、模板、会员、任务、后台管理接口。

3. 应用服务层
- 负责项目管理、章节编排、段落改写、模板应用、会员扣减、内容安全校验。
- 不直接依赖具体模型 SDK，而通过 Agent Runner 抽象调用 AI 能力。

4. 编排层
- 负责章节级状态机与自动续写任务状态机。
- 保证单次任务最多 10 章，章节间严格串行，避免跨章上下文错乱。
- 控制每章固定流水线：`走向生成 -> 主编评分选优 -> 正文生成 -> 读者评分 -> 回流/通过 -> 记忆更新`。

5. 领域与存储层
- 使用关系型数据库存储项目、章节、候选方案、正文版本、记忆实体、任务记录、会员额度、订单、审计日志索引。
- 正文与长文本可先存数据库文本字段；后续规模增大时可迁移到对象存储，但不改变业务主键关系。

6. AI 适配层
- 统一封装故事编辑 Agent、资深主编 Agent、写手 Agent、资深读者 Agent。
- 统一处理 prompt 拼装、模型调用、重试、结构化结果解析、日志与指标。

建议部署单元：
- `apps/api`: FastAPI 服务，处理同步 API。
- `apps/worker`: 后台任务 Worker，执行章节流水线、回流、记忆更新、审计写入。
- `packages/domain`: 领域模型、状态机、规则与服务。
- `packages/integrations`: 模型调用、队列、存储、日志等基础设施适配。

## 模块划分
### 1. 项目中心模块
- 创建小说项目，记录题材、篇幅、模板、模式。
- 管理项目基础元信息与当前章节推进位置。
- 作为后续章节、模板、记忆、额度校验的聚合根。

### 2. 大纲与章节编辑模块
- 管理总纲、章节目标、章节大纲、章节正文。
- 支持与故事编辑 Agent 的近实时对话式大纲优化。
- 支持人工确认模式下的逐章确认与入库。

### 3. 模板模块
- 支持系统模板与用户自定义模板。
- 模板字段至少覆盖题材约束、世界观/角色预设、大纲框架、文风约束。
- 后台支持模板审核、发布、下架、版本更新。

### 4. 章节编排模块
- 负责单章生成与自动续写任务。
- 控制每章必须生成 3 个走向方案。
- 控制正文评分阈值、最多 5 次回流、最高分版本保留规则。
- 维护每章与整批任务的状态展示数据。

### 5. Agent 协作模块
- 故事编辑 Agent：输出 3 个剧情走向候选。
- 资深主编 Agent：对候选走向评分并自动选出最高分方案。
- 写手 Agent：基于选中走向与记忆上下文生成正文。
- 资深读者 Agent：对正文评分、给出问题说明，并决定是否回流。

### 6. 结构化记忆模块
- 管理全书摘要、角色卡、世界观设定、已发生事件摘要、章节状态。
- 为章节生成、自动续写、段落重写/扩写统一拼装上下文。
- 在章节最终输出后更新记忆快照，供下一章消费。

### 7. 段落编辑模块
- 支持任意段落的重写、扩写。
- 输出差异对比和上下文一致性检查结果。
- 不改变章节主流程规则，只作为章节正文上的局部 AI 编辑能力。

### 8. 会员与配额模块
- 管理免费章节额度、月度额度、套餐规则、订单、手动调额。
- 在生成、回流、局部改写前执行额度校验与扣减。
- 为后台提供套餐配置与配额审计能力。

### 9. 内容安全与审计模块
- 执行敏感题材过滤、版权声明展示、数据留存。
- 记录核心操作日志：任务创建、Agent 输出、评分结果、回流次数、人工确认、后台调额。

## API 设计
首期建议采用 REST 风格 API，统一响应格式：

```json
{
  "code": "OK",
  "message": "success",
  "data": {}
}
```

错误响应必须返回明确错误码和错误信息，不允许 silent fail。

### 项目与大纲 API
- `POST /api/v1/projects`
  - 创建项目。
  - 入参：标题、题材、篇幅、模板 ID、创作模式。

- `GET /api/v1/projects/{project_id}`
  - 查询项目详情、章节进度、当前模式、记忆摘要。

- `PATCH /api/v1/projects/{project_id}`
  - 修改项目元信息。

- `PUT /api/v1/projects/{project_id}/outline`
  - 保存总纲、文风约束、全书目标。

- `PUT /api/v1/projects/{project_id}/chapters/{chapter_id}/brief`
  - 保存章节目标与章节大纲。

### 章节生成与任务 API
- `POST /api/v1/projects/{project_id}/chapters/{chapter_id}/generate`
  - 发起单章生成。
  - 支持人工确认模式下的单章流程。

- `POST /api/v1/projects/{project_id}/autopilot-runs`
  - 发起自动续写任务。
  - 必须校验 `target_chapter_count <= 10`。

- `GET /api/v1/generation-runs/{run_id}`
  - 查询任务状态、当前章节、已完成章节、评分、回流次数、建议人工复核标记。

- `GET /api/v1/chapters/{chapter_id}`
  - 查询章节详情，包括 3 个走向方案、选中方案、正文版本列表、最高分版本。

- `POST /api/v1/chapters/{chapter_id}/outline-options/{option_id}/confirm`
  - 人工确认模式下手动确认走向方案。

- `POST /api/v1/chapters/{chapter_id}/finalize`
  - 人工确认最终正文版本入库。

### 段落级编辑 API
- `POST /api/v1/chapters/{chapter_id}/paragraph-edits`
  - 入参：段落范围、操作类型（`rewrite`/`expand`）、补充要求。
  - 出参：新文本、差异对比、一致性检查结果。

### 模板与后台 API
- `GET /api/v1/templates`
- `POST /api/v1/templates`
- `PATCH /api/v1/templates/{template_id}`
- `POST /api/v1/admin/templates/{template_id}/review`
- `GET /api/v1/admin/membership-plans`
- `POST /api/v1/admin/membership-plans`
- `POST /api/v1/admin/quotas/adjustments`
- `GET /api/v1/admin/orders`
- `GET /api/v1/admin/logs`

### 错误码约定
- `400 Bad Request`: 参数不合法，例如续写章数大于 10。
- `403 Forbidden`: 无访问权限或无后台权限。
- `404 Not Found`: 项目、章节、模板、任务不存在。
- `409 Conflict`: 状态冲突，例如章节仍在处理中却重复发起任务。
- `422 Unprocessable Entity`: 缺少生成前置条件，例如未配置总纲。
- `429 Too Many Requests`: 会员额度不足或达到频率限制。
- `503 Service Unavailable`: Agent 调用失败或任务系统不可用。

## 状态流/权限流
### 章节状态流
建议章节状态最少包含：
- `draft`
- `outline_generating`
- `outline_scored`
- `draft_generating`
- `reviewing`
- `revision_pending`
- `completed`
- `manual_review_required`
- `failed`

典型状态流：

1. 创建章节后进入 `draft`
2. 故事编辑 Agent 生成 3 个走向时进入 `outline_generating`
3. 资深主编 Agent 完成评分并选优后进入 `outline_scored`
4. 写手 Agent 生成正文时进入 `draft_generating`
5. 资深读者 Agent 评分时进入 `reviewing`
6. 若分数 `< 8.0` 且回流次数未达 5，则进入 `revision_pending` 并再次回到 `draft_generating`
7. 若分数 `>= 8.0`，进入 `completed`
8. 若达到最大回流次数仍未达标，则输出最高分版本并进入 `manual_review_required`

### 自动续写任务状态流
- `pending`
- `running`
- `paused`
- `completed`
- `partial_completed`
- `failed`

规则：
- 任务创建时校验章节额度、项目前置条件、目标章数上限。
- 自动任务逐章执行；当前章未结束，下一章不可开始。
- 任一章节进入 `manual_review_required` 不必终止整批任务，但必须在任务结果中显式标记。

### 权限流
首期采用基础角色划分：
- 创作者用户
  - 管理自己的项目、模板、章节、任务、段落编辑。
- 后台运营用户
  - 审核模板、管理套餐、手动调额、查看订单与日志。
- 系统任务身份
  - 执行异步编排、调用 Agent、写任务日志，不直接暴露给前端。

权限原则：
- 项目、章节、记忆、私有模板按所属用户隔离。
- 后台接口必须与创作接口分离。
- 手动调额、模板审核、日志查看均需后台权限。

## 模板/会员/AI协作（如适用）
### 模板策略
- 模板分为官方模板与用户私有模板。
- 模板实例化时写入项目快照，避免后续模板更新影响已创建项目。
- 模板至少提供题材约束、世界观预设、角色预设、章节框架、文风规则。

### 会员策略
- 首期按章节生成额度、重写额度、自动续写额度管理。
- 回流重写与段落级重写应计入可追踪额度流水，便于后台核查。
- 手动调额必须写审计日志。

### AI 协作策略
固定协作链路如下：

1. 故事编辑 Agent
- 输入：章节目标、上文摘要、角色卡、世界观约束、已发生事件。
- 输出：3 个走向方案，每个包含核心冲突、关键事件、结尾钩子。

2. 资深主编 Agent
- 输入：3 个走向方案。
- 输出：每个方案的三维评分、说明、被选中的最高分方案。

3. 写手 Agent
- 输入：被选中走向方案、结构化记忆、文风约束。
- 输出：章节正文。

4. 资深读者 Agent
- 输入：章节正文。
- 输出：可读性、情绪张力、连贯性评分，问题说明，是否回流。

关键约束：
- 主编评分与读者评分都使用 10 分制。
- 正文通过阈值固定为 `8.0`。
- 每章最多回流 5 次。
- 5 次后仍未达标时，不继续追加生成，直接输出历史最高分版本并标记人工复核建议。

## 错误处理
错误处理必须遵循 `docs/engineering-standards.md`：
- API 必须有错误返回。
- 不允许 silent fail。
- 核心流程必须有日志。

设计要求如下：

### 同步 API 错误
- 统一返回业务错误码、可读错误信息、请求追踪 ID。
- 参数错误在 API 层直接拒绝。
- 权限、额度、状态冲突在应用服务层返回明确错误。

### 异步任务错误
- 单次 Agent 调用失败时允许有限重试，但不得突破章节流程约束。
- 若某一步失败，必须写入 `ChapterTaskLog` 并更新任务状态。
- 自动续写任务出错时应保留已完成章节结果，不回滚已成功章节。

### Agent 输出错误
- 对结构化输出执行 schema 校验。
- 若故事走向数量不是 3、评分缺失、正文为空等，视为任务失败并记录日志。
- 不允许以默认空内容代替失败结果。

### 内容安全错误
- 命中敏感内容策略时拒绝继续生成，并将原因反馈给用户或后台。
- 审计日志必须记录触发规则与上下文摘要。

## 可扩展性
首期虽聚焦章节工坊，但架构需为后续能力预留扩展点。

扩展策略如下：
- Agent Runner 抽象化，允许未来新增配图 Agent、社区审校 Agent，而不重写章节编排核心。
- 结构化记忆拆分为独立实体，后续可增加伏笔卡、关系图、卷册层级。
- `GenerationRun` 与 `ChapterTaskLog` 为后续更复杂队列调度和监控面板提供基础。
- 模板采用版本化与项目快照机制，便于后续扩展模板市场。
- 正文存储与数据库主数据解耦，未来可平滑迁移到对象存储或检索索引。
- API 保持资源化设计，后续接入移动端或专门管理端时无需重构领域模型。

明确限制：
- 首期不为了“扩展性”引入复杂微服务拆分。
- 首期不实现多章并行写作，以免破坏连贯性要求。

## 非功能约束
### 技术约束
- 后端必须使用 FastAPI。
- 测试框架必须使用 pytest。
- 不允许硬编码 secrets、token、数据库密码。
- 仅在允许路径内产出设计文档与代码变更。

### 一致性与质量约束
- 自动续写必须串行执行，优先保证上下文一致性而非吞吐。
- 章节评分、回流次数、最高分版本保留规则必须可审计、可复现。
- 段落级改写必须执行上下文一致性检查与差异展示。

### 性能约束
- API 层避免长时间阻塞；长耗时生成必须走异步任务。
- 任务状态查询应支持前端轮询，不要求首期实现复杂实时推送。

### 可靠性约束
- 任何章节任务失败都必须保留中间结果和日志，便于人工复核。
- 已完成章节不得因后续章节失败被回滚。

### 安全与合规约束
- 项目数据、私有模板、会员信息按用户隔离。
- 后台操作必须保留审计记录。
- 必须支持敏感题材过滤、版权声明和数据留存要求。

### 可观测性约束
- 核心流程必须记录结构化日志。
- 至少跟踪任务开始/结束、Agent 调用耗时、评分结果、回流次数、失败原因、额度扣减。
