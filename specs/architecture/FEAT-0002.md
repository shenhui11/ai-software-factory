# FEAT-0002 Architecture

## 1. 模块边界

### 1.1 总体架构

`FEAT-0002` 采用现有单体式 FastAPI 后端扩展实现，在当前 `apps/backend` 分层结构内新增会员域能力，不引入独立微服务、消息队列或复杂规则引擎。首版重点是建立“成长积分型会员体系”的最小闭环：任务触发、积分记账、等级变更、权益生效、运营配置、日志追踪。

推荐继续沿用四层结构：

- Presentation Layer：FastAPI 路由、请求校验、响应序列化、统一错误结构。
- Application Layer：会员用例编排，负责签到、任务完成、积分发放、升级计算、奖励发放、配置发布。
- Domain Layer：会员等级、积分账户、任务规则、权益映射、防重复约束等核心模型与业务规则。
- Infrastructure Layer：持久化、配置加载、日志、审计事件、时间源、幂等键存储。

### 1.2 领域模块拆分

- `member_profile`：用户会员档案，维护当前等级、会员状态、成长值、最近升级时间。
- `points_account`：积分账户，维护可用积分、累计获得积分、累计消耗积分。
- `task_center`：成长任务定义与用户任务完成状态，首版支持每日签到、指定行为、连续活跃。
- `points_ledger`：积分流水，记录来源、变更类型、数量、关联任务、关联请求、操作时间。
- `level_engine`：等级门槛判断、升级计算、下一等级进度计算。
- `benefit_entitlement`：等级权益映射与权益生效判断，首版支持至少一种真实权益落地。
- `reward_dispatch`：奖励发放编排，负责把升级奖励或任务奖励转换为可感知结果。
- `ops_config`：运营配置管理，维护任务开关、积分值、等级门槛、权益映射，并支持版本化发布。
- `risk_control`：最小必要防刷限制，处理重复签到、短时间高频触发、幂等校验。
- `member_observability`：核心事件日志、审计日志、指标埋点。

### 1.3 关键边界约束

- `task_center` 只负责判断“任务是否完成”，不直接写积分余额。
- `points_account` 不直接决定等级，等级变化必须经过 `level_engine`。
- `benefit_entitlement` 只负责权益判断与映射，不承载任务规则。
- `ops_config` 是任务积分值、等级门槛、权益映射的唯一配置来源，禁止在代码中硬编码业务参数。
- `risk_control` 必须在积分发放前执行，防止重复请求造成重复记账。
- `points_ledger` 是积分变更的审计事实来源，任何余额变化都必须先有流水。
- MVP 保持同步事务式处理，不引入异步事件总线；内部仍可用“领域事件对象”组织代码，便于后续演进。

## 2. API 设计

### 2.1 API 约定

- 路径前缀：`/api/v1`
- 风格：REST JSON API
- 鉴权：沿用现有登录态或鉴权中间件，用户接口默认要求已登录
- 错误结构统一为：

```json
{
  "error": {
    "code": "string_code",
    "message": "human_readable_message",
    "details": {}
  },
  "request_id": "uuid"
}
```

### 2.2 用户侧接口

- `GET /api/v1/members/me`
  - 返回当前会员档案。
  - 响应包含：`membership_status`、`current_level`、`growth_value`、`next_level`、`growth_to_next_level`、`points_balance`。

- `GET /api/v1/members/me/dashboard`
  - 返回会员面板聚合数据。
  - 响应包含：当前等级、成长进度、积分余额、已解锁权益、待完成任务、最近奖励。

- `GET /api/v1/members/me/tasks`
  - 返回当前可见任务列表。
  - 每个任务包含：`task_code`、`title`、`task_type`、`reward_points`、`status`、`progress`、`daily_reset_at`。

- `POST /api/v1/members/me/check-ins`
  - 执行每日签到。
  - 幂等约束：同一用户同一自然日只允许成功一次。
  - 返回：是否签到成功、发放积分、当前等级、是否升级。

- `POST /api/v1/members/me/tasks/{task_code}/complete`
  - 手动提交首版允许用户触发的任务完成。
  - 适用于需要显式提交的任务类型；系统行为型任务也可由内部应用服务调用同一用例。
  - 返回：任务完成结果、积分变更、奖励结果、升级结果。

- `GET /api/v1/members/me/points/ledger`
  - 查询积分流水。
  - 支持参数：`cursor`、`limit`、`change_type`。

- `GET /api/v1/members/me/rewards`
  - 查询已获得奖励记录。

### 2.3 运营侧接口

- `GET /api/v1/admin/member-config`
  - 获取当前生效的会员配置快照。

- `PUT /api/v1/admin/member-config/tasks`
  - 更新任务配置。
  - 请求体包含：任务开关、任务积分值、每日上限、连续活跃规则。

- `PUT /api/v1/admin/member-config/levels`
  - 更新等级门槛配置。
  - 请求体包含：等级编号、等级名称、成长值门槛、展示文案。

- `PUT /api/v1/admin/member-config/benefits`
  - 更新等级权益映射。
  - 请求体包含：等级与权益编码映射、额度类权益参数、标识类权益开关。

- `POST /api/v1/admin/member-config:publish`
  - 发布配置版本。
  - 返回配置版本号与发布时间。

- `GET /api/v1/admin/members/{user_id}`
  - 查询指定用户会员详情与最近流水，便于客服和运营排查。

### 2.4 内部服务接口

### 3.2 数据一致性规则

- 积分余额变更必须与 `PointsLedgerEntry` 同事务提交。
- 等级升级与奖励发放必须基于同一业务请求上下文，避免“已升级但未发奖”。
- 配置变更采用版本号读取，业务执行时固化所使用的 `config_version`，便于追溯。
- 同一任务完成请求必须使用 `request_id` 或业务幂等键，避免重复积分。

## 4. 权限与安全

### 4.1 权限模型

- 普通登录用户：只能访问自己的会员面板、任务、积分流水、奖励记录。
- 运营人员：可查看和发布会员配置，可查询指定用户会员信息。
- 平台管理员：可执行人工修正、回滚、补发等高权限操作；该能力不应默认开放给运营角色。

### 4.2 安全要求

- 所有用户侧接口必须校验当前登录用户与目标 `user_id` 一致，禁止越权读取他人账户信息。
- 运营配置接口必须增加角色校验和操作审计。
- 禁止硬编码积分值、等级门槛、奖励参数、密钥、数据库密码。
- 所有人工调整积分或奖励的接口必须记录操作者身份、原因和请求号。
- 对签到和任务完成接口实施幂等控制、频率限制和基础反重放校验。
- 对奖励发放失败场景返回明确错误，不允许 silent fail。

## 5. 可观测性

### 5.1 日志

核心流程必须写结构化日志，至少覆盖：

- 用户签到请求
- 任务完成判定
- 积分发放成功与拒绝
- 等级升级
- 奖励发放
- 运营配置发布
- 人工修正操作

日志字段建议统一包含：`request_id`、`user_id`、`task_code`、`level_code`、`points_delta`、`result`、`error_code`。

### 5.2 指标

建议暴露以下基础指标：

- `member_check_in_success_total`
- `member_task_completion_total`
- `member_points_earned_total`
- `member_level_up_total`
- `member_reward_issue_total`
- `member_risk_reject_total`
- `member_config_publish_total`

### 5.3 审计与追踪

- 核心会员动作写入 `MemberAuditEvent`，用于事后追溯。
- API 层为每次请求生成或透传 `request_id`，错误返回中必须带回。
- MVP 可先使用应用日志 + 数据库审计表，不强制接入完整分布式 tracing。

## 6. 部署说明

- 继续沿用现有单体后端部署方式，通过 `uvicorn apps.backend.main:app --host 0.0.0.0 --port 8000` 启动。
- 会员能力作为现有 FastAPI 应用的新路由与服务模块部署，不新增独立进程。
- 需要新增会员相关数据表迁移，并在部署前完成配置初始化：
  - 任务配置
  - 等级门槛配置
  - 权益映射配置
- 运行时配置通过环境变量和数据库配置表提供，不通过代码常量写死业务参数。
- 发布顺序建议为：数据库迁移 -> 初始化默认配置 -> 部署应用 -> 验证健康检查与配置读取。

## 7. 风险与技术债

- 首版采用同步事务式处理，简单但会把“任务判定 + 记账 + 升级 + 发奖”耦合在一次请求中；后续流量上升时可能需要拆为异步事件流。
- 当前需求没有明确现成“可感知权益”，若首版权益仅停留在标识展示，会员价值感可能不足。
- 若未来接入更多任务类型，固定任务模型可能快速膨胀，届时需要演进为更清晰的规则定义层。
- 基础防刷仅覆盖重复签到和高频触发，无法应对更复杂作弊行为，后续可能需要补充设备、IP、行为模式分析。
- 运营配置发布为 MVP 的关键能力，但若缺少版本回滚和变更预览，容易导致误配置直接影响用户积分与等级。
- 积分流水是财务式审计数据，后续如引入积分消耗、积分商城或付费会员，需要进一步强化对账和补偿机制。
