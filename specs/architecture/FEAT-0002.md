# FEAT-0002 Architecture

## 1. 模块边界

### 1.1 总体架构

`FEAT-0002` 采用现有单体式 FastAPI 后端扩展实现，不新增独立微服务、消息队列或复杂规则引擎。首版目标是以最小复杂度建立“成长积分型会员体系”闭环，覆盖任务触发、积分记账、等级升级、权益映射、运营配置和事件追踪。

推荐继续沿用四层结构：

- Presentation Layer：FastAPI 路由、请求鉴权、参数校验、响应序列化、统一错误结构。
- Application Layer：会员用例编排，负责签到、任务完成、积分发放、等级评估、奖励发放、配置发布。
- Domain Layer：会员档案、积分账户、任务规则、等级规则、权益映射、防重复约束等核心业务模型和规则。
- Infrastructure Layer：数据库仓储、配置加载、日志、审计事件、时间源、幂等键和监控埋点。

### 1.2 领域模块拆分

- `member_profile`：会员档案域，维护会员状态、当前等级、成长值、最近升级时间。
- `points_account`：积分账户域，维护可用积分、累计获得积分、累计消耗积分。
- `task_center`：任务域，管理任务定义、用户任务进度、签到和指定行为的完成状态。
- `points_ledger`：积分流水域，记录每次积分变更的来源、数量、余额快照和请求上下文。
- `level_engine`：等级引擎，负责门槛判断、升级计算、下一等级进度计算。
- `benefit_entitlement`：权益域，负责等级到权益的映射、用户已解锁权益解析和生效判断。
- `reward_dispatch`：奖励发放域，负责把任务奖励或升级奖励落地到可感知结果。
- `ops_config`：运营配置域，维护任务开关、积分值、等级门槛和权益映射的版本化配置。
- `risk_control`：基础风控域，负责重复签到、重复任务提交、短时高频刷分的最小必要限制。
- `member_observability`：可观测性域，统一记录核心会员事件、业务日志、指标和审计信息。

### 1.3 边界约束

- `task_center` 只负责判断任务是否达成，不直接修改积分余额。
- `points_account` 只负责积分账务，不直接决定等级变化。
- `level_engine` 是等级升级和进度计算的唯一入口，禁止在路由或仓储层绕过。
- `benefit_entitlement` 只负责权益解析和命中判断，不承载积分或任务规则。
- `ops_config` 是任务积分值、等级门槛和权益映射的唯一配置来源，禁止在代码中硬编码业务参数。
- `risk_control` 必须在积分记账前执行，防止重复请求造成重复发分。
- `points_ledger` 是积分变更的审计事实来源，任何积分余额变化都必须伴随流水记录。
- MVP 采用同步事务式处理，避免为了首版可用性引入异步一致性复杂度。

## 2. API 设计

### 2.1 通用约定

- 路径前缀：`/api/v1`
- 风格：REST JSON API
- 鉴权：沿用现有登录态或鉴权中间件，用户接口默认要求已登录
- 错误结构统一为：

```json
{
  "error": {
    "code": "string_code",
    "message": "human_readable_message",
    "details": {}
  },
  "request_id": "uuid"
}
```

### 2.2 用户侧接口

#### `GET /api/v1/members/me`

返回当前会员档案摘要。

响应字段建议包含：

- `membership_status`
- `current_level`
- `growth_value`
- `next_level`
- `growth_to_next_level`
- `points_balance`
- `unlocked_benefits`

#### `GET /api/v1/members/me/dashboard`

返回会员面板聚合数据。

响应字段建议包含：

- 当前等级和成长进度
- 积分余额和累计获得积分
- 已解锁权益
- 待完成任务
- 最近奖励记录

#### `GET /api/v1/members/me/tasks`

返回当前用户可见任务列表。

每个任务字段建议包含：

- `task_code`
- `title`
- `task_type`
- `reward_points`
- `status`
- `progress`
- `daily_reset_at`

#### `POST /api/v1/members/me/check-ins`

执行每日签到。

约束：

- 同一用户同一自然日只允许成功一次
- 必须经过幂等校验和基础频控

响应字段建议包含：

- `success`
- `granted_points`
- `points_balance`
- `current_level`
- `level_up`
- `unlocked_benefits`

#### `POST /api/v1/members/me/tasks/{task_code}/complete`

用于显式提交可手动完成的任务；系统行为型任务也可在应用层复用相同用例。

响应字段建议包含：

- `task_code`
- `completion_status`
- `granted_points`
- `points_balance`
- `level_up`
- `reward_result`

#### `GET /api/v1/members/me/points/ledger`

查询积分流水。

支持参数：

- `cursor`
- `limit`
- `change_type`

#### `GET /api/v1/members/me/rewards`

查询用户已获得奖励记录。

### 2.3 运营侧接口

#### `GET /api/v1/admin/member-config`

获取当前生效的会员配置快照。

#### `PUT /api/v1/admin/member-config/tasks`

更新任务配置。

请求体应支持：

- 任务开关
- 任务积分值
- 每日上限
- 连续活跃规则

#### `PUT /api/v1/admin/member-config/levels`

更新等级门槛配置。

请求体应支持：

- 等级编号
- 等级名称
- 成长值门槛
- 展示文案

#### `PUT /api/v1/admin/member-config/benefits`

更新等级权益映射。

请求体应支持：

- 等级与权益编码映射
- 权益启停状态
- 额度类权益参数

#### `POST /api/v1/admin/member-config:publish`

发布配置版本，返回配置版本号和发布时间。

#### `GET /api/v1/admin/members/{user_id}`

查询指定用户的会员详情、任务状态和最近流水，用于运营排查。

### 2.4 应用层内部服务

- `MemberDashboardService`
- `CheckInService`
- `TaskCompletionService`
- `PointsAccountingService`
- `LevelEvaluationService`
- `BenefitResolutionService`
- `RewardDispatchService`
- `MemberConfigService`
- `MemberRiskControlService`

### 3.2 一致性与约束

- 积分余额变更必须与 `PointsLedgerEntry` 同事务提交。
- 任务完成、积分发放、等级评估应在单次业务事务内完成，避免出现“任务已完成但未发分”。
- 等级升级与奖励发放必须共享同一请求上下文，避免“已升级但未发奖”。
- 配置变更采用版本号管理，业务执行时记录所用 `config_version`，保证可追溯。
- 同一任务完成请求必须使用 `request_id` 或等价幂等键，避免重复积分。

## 4. 权限与安全

### 4.1 权限边界

- 普通登录用户只能访问自己的会员档案、任务、积分流水和奖励记录。
- 运营接口必须与普通用户接口分离，要求管理员或具备运营配置权限的后台角色访问。
- `GET /api/v1/admin/members/{user_id}` 仅用于排查和客服场景，不允许无边界导出用户数据。

### 4.2 安全控制

- 所有接口必须复用现有鉴权机制，不允许匿名修改会员状态或运营配置。
- 运营配置变更必须记录操作者、变更前后内容、发布时间和配置版本。
- 禁止硬编码 token、密码或业务敏感配置；阈值和开关统一走配置。
- 对签到、任务完成、手工调整类接口使用幂等键和频率限制，降低重复提交和刷分风险。
- 对关键字段做服务端校验，禁止由客户端直接提交积分值、等级值或权益状态。
- 日志和审计事件中不得输出敏感认证信息；用户标识应按现有日志规范处理。

## 5. 可观测性

### 5.1 日志

核心流程必须记录结构化日志，至少覆盖：

- 用户签到
- 任务完成
- 积分发放
- 等级升级
- 奖励发放
- 运营配置发布
- 风控拦截

每条日志应包含：

- `request_id`
- `user_id` 或后台操作者标识
- `event_type`
- `result`
- `config_version`

### 5.2 指标

建议首版增加以下业务指标：

- `member_check_in_success_total`
- `member_task_completion_total`
- `member_points_granted_total`
- `member_level_up_total`
- `member_reward_issued_total`
- `member_risk_block_total`
- `member_config_publish_total`

### 5.3 审计与追踪

- 使用 `request_id` 串联路由层、应用层、账务流水和审计事件。
- 对积分发放、等级升级、配置发布保留审计事件，满足后续排障和运营复盘。
- 当接口失败时，必须返回统一错误结构，并记录明确错误码，不允许 silent fail。

## 6. 部署说明

- 首版作为现有 FastAPI 应用内新增模块部署，不新增独立服务。
- 数据库需要新增会员相关表：会员档案、积分账户、任务配置、任务记录、积分流水、等级配置、权益配置、奖励记录、审计事件。
- 配置发布能力依赖现有后台管理入口；若后台尚未存在，应在现有应用内部扩展，不单独创建新系统。
- 新环境启动方式应继续沿用项目现有后端启动命令，避免为该功能引入额外部署单元。
- 上线前应完成数据库迁移、默认等级配置、默认任务配置和至少一种真实权益配置。

## 7. 风险与技术债

### 7.1 主要风险

- 原始需求只说明“添加会员系统”，如果业务方实际预期是首版直接支持付费会员，当前架构无法覆盖收入相关链路。
- 如果首版无法提供用户可感知的真实权益，会员系统可能退化为仅展示积分和等级，验证价值有限。
- 任务门槛和积分值若配置不合理，可能导致升级过快、升级过慢或用户对公平性产生质疑。
- 当前仅做基础风控，无法覆盖复杂作弊场景，存在被刷分的现实风险。

### 7.2 技术债与后续演进

- 当前采用同步事务式处理，后续若奖励发放或行为触发来源增多，可能需要演进为事件驱动模型。
- 当前配置模型偏静态，后续若出现多条件、多周期任务，需引入更强的规则表达能力。
- 当前权益模型适合 MVP，后续如扩展到优惠券、额度包或付费订阅，需细化权益生命周期和核销模型。
- 当前审计和指标以基础追踪为主，后续若要评估留存和转化效果，还需要补充更完整的数据分析链路。
