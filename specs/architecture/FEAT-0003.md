# FEAT-0003 Architecture

## 1. 模块边界

### 1.1 总体架构

`FEAT-0003` 采用现有单体式 FastAPI 后端增量扩展实现，在当前应用内增加“模板资源库平台”能力，不引入独立模板微服务、消息队列或复杂工作流引擎。首版目标是完成系统模板治理、用户模板沉淀、模板复制复用和 AI 单轮生成草稿的最小闭环。

推荐沿用四层结构：

- Presentation Layer：FastAPI 路由层，负责请求校验、响应序列化、统一错误结构、鉴权结果接入。
- Application Layer：模板用例编排层，负责模板查询、系统模板管理、用户模板 CRUD、模板复制、AI 草稿生成。
- Domain Layer：模板、分类、标签、权限边界、状态流转等核心模型与业务规则。
- Infrastructure Layer：数据库持久化、AI 网关、日志、配置、审计与时间源。

### 1.2 领域模块拆分

- `template_catalog`：模板聚合查询，负责前台模板列表、模板详情、统一筛选与展示。
- `system_template_admin`：系统模板后台管理，负责创建、编辑、发布、下线、删除。
- `user_template`：用户模板管理，负责个人模板创建、编辑、删除、查看。
- `template_clone`：模板复制能力，负责把系统模板复制为用户模板，并记录来源关系。
- `template_taxonomy`：分类与标签管理，负责分类、标签维护及模板归类。
- `template_generation`：AI 模板草稿生成，负责接收一句话需求，调用模型并返回可编辑草稿。
- `template_access_control`：权限判断与资源归属校验，统一处理管理员权限、模板所有权与可见范围。
- `template_observability`：模板域结构化日志、审计事件、基础指标。

### 1.3 关键边界约束

- `system_template_admin` 只能操作 `template_type=system` 的模板，且必须经过管理员鉴权。
- `user_template` 只能操作当前登录用户拥有的 `template_type=user` 模板，禁止跨用户访问。
- `template_clone` 是系统模板转个人模板的唯一入口，普通用户不得直接修改系统模板原文。
- `template_generation` 只负责返回草稿，不直接自动发布、不直接写入系统模板。
- `template_taxonomy` 维护的是模板分类与标签元数据，不承载模板权限逻辑。
- MVP 采用同步请求处理，不引入异步任务中心；若 AI 生成响应过慢，后续再演进为异步草稿任务。
- 模板“使用”为把模板内容返回给现有内容/任务创建入口作为起始输入，不在本期架构中扩展成流程执行能力。

## 2. API 设计

### 2.1 API 约定

- 路径前缀：`/api/v1`
- 风格：REST JSON API
- 鉴权：沿用现有登录态；管理员接口要求具备后台管理角色
- 错误结构统一为：

```json
{
  "error": {
    "code": "string_code",
    "message": "human_readable_message",
    "details": {}
  },
  "request_id": "uuid"
}
```

### 2.2 前台用户接口

- `GET /api/v1/templates`
  - 查询模板列表。
  - 支持参数：`scope=system|mine|all`、`keyword`、`category_id`、`tag`、`page`、`page_size`。
  - 默认仅返回已发布系统模板和当前用户自己的模板。

- `GET /api/v1/templates/{template_id}`
  - 查询模板详情。
  - 系统模板仅允许查看已发布或已下线但具备管理权限的记录；用户模板仅允许所有者查看。

- `POST /api/v1/templates/user`
  - 创建用户模板。
  - 请求体包含：`name`、`description`、`content`、`category_id`、`tag_ids`。

- `PATCH /api/v1/templates/user/{template_id}`
  - 更新当前用户的用户模板。

- `DELETE /api/v1/templates/user/{template_id}`
  - 删除当前用户的用户模板。

- `POST /api/v1/templates/{template_id}:clone`
  - 基于系统模板复制生成个人模板。
  - 返回新创建的用户模板详情。

- `POST /api/v1/templates/drafts:generate`
  - 根据一句话需求生成模板草稿。
  - 请求体包含：`prompt`、可选 `context`。
  - 返回：`name`、`description`、`content`、建议分类/标签、生成说明。

- `POST /api/v1/templates/{template_id}:use`
  - 获取模板作为创建内容或任务的起始载荷。
  - 返回标准化模板内容快照，不修改模板本身。

### 2.3 管理后台接口

- `GET /api/v1/admin/templates`
  - 查询系统模板列表。
  - 支持参数：`status=draft|published|offline`、`keyword`、`category_id`、`tag_id`、`page`、`page_size`。

- `POST /api/v1/admin/templates`
  - 创建系统模板。
  - 请求体包含：`name`、`description`、`content`、`category_id`、`tag_ids`、`status`。

- `PATCH /api/v1/admin/templates/{template_id}`
  - 编辑系统模板基础信息与内容。

- `POST /api/v1/admin/templates/{template_id}:publish`
  - 发布系统模板。

- `POST /api/v1/admin/templates/{template_id}:offline`
  - 下线系统模板。

- `DELETE /api/v1/admin/templates/{template_id}`
  - 删除系统模板。

- `GET /api/v1/admin/template-categories`
  - 查询分类列表。

- `POST /api/v1/admin/template-categories`
  - 创建分类。

- `PATCH /api/v1/admin/template-categories/{category_id}`
  - 更新分类。

- `DELETE /api/v1/admin/template-categories/{category_id}`
  - 删除分类。

- `GET /api/v1/admin/template-tags`
  - 查询标签列表。

- `POST /api/v1/admin/template-tags`
  - 创建标签。

- `PATCH /api/v1/admin/template-tags/{tag_id}`
  - 更新标签。

- `DELETE /api/v1/admin/template-tags/{tag_id}`
  - 删除标签。

### 2.4 内部服务接口

- `TemplateQueryService`
- `SystemTemplateAdminService`
- `UserTemplateService`
- `TemplateCloneService`
- `TemplateGenerationService`
- `TemplateTaxonomyService`
- `TemplatePermissionService`
- `TemplateUsageService`
- `AiGateway`

## 3. 数据模型

### 3.1 核心实体

#### Template

- `id`
- `template_type`
- `status`
- `name`
- `description`
- `content`
- `category_id`
- `creator_user_id`
- `owner_user_id`
- `source_template_id`
- `ai_generated`
- `created_at`
- `updated_at`

说明：

- `template_type` 取值至少包括 `system`、`user`。
- `status` 首版至少支持 `draft`、`published`、`offline`。
- 对系统模板，`owner_user_id` 为空或等于平台保留主体；对用户模板，`owner_user_id` 必须等于创建者。
- `source_template_id` 用于标记用户模板是否由系统模板复制而来。

- `event_payload`

### 3.2 数据一致性规则

- 系统模板发布前必须满足必填字段校验通过，且分类必须有效。
- 模板删除仅删除目标模板本身，不级联删除由其复制而来的用户模板。
- 模板标签关系与模板主体更新应在同一事务内提交，避免标签部分写入。
- 模板复制时应固化源模板内容快照到新用户模板，后续系统模板更新不反向影响已复制的用户模板。
- AI 生成记录应与用户保存模板解耦，生成成功不代表模板已入库。

## 4. 权限与安全

### 4.1 权限模型

- 普通登录用户：可查看已发布系统模板，可管理自己的用户模板，可调用 AI 生成草稿。
- 管理员：可管理系统模板、分类和标签，可查看后台筛选结果与模板状态。
- 非模板拥有者：不得查看、编辑、删除其他用户的用户模板。

### 4.2 安全要求

- 所有用户模板接口必须基于当前登录用户做资源归属校验，禁止通过 `template_id` 越权访问。
- 所有后台接口必须做角色校验，并记录操作审计日志。
- AI 生成接口不得硬编码模型密钥或敏感配置，统一通过环境变量或安全配置中心注入。
- 模板内容输入必须做基础长度限制和空值校验，防止异常大请求拖垮服务。
- 模板详情输出需要按权限过滤，禁止把未发布系统模板暴露给普通用户。
- 删除、发布、下线等后台操作必须返回明确错误信息，不允许 silent fail。
- 若模板内容最终会流向前端富文本展示层，渲染侧需要补充 XSS 过滤或安全转义；本期后端至少保证按纯文本/安全字符串接口返回。

## 5. 可观测性

### 5.1 日志

核心流程必须写结构化日志，至少覆盖：

- 系统模板创建、编辑、发布、下线、删除
- 用户模板创建、编辑、删除
- 模板复制
- 模板列表查询与详情读取失败
- AI 模板草稿生成成功与失败
- 分类与标签管理操作

建议统一日志字段：

- `request_id`
- `user_id`
- `template_id`
- `template_type`
- `status`
- `event_type`
- `result`
- `error_code`

### 5.2 指标

建议暴露以下基础指标：

- `template_query_total`
- `system_template_publish_total`
- `system_template_offline_total`
- `user_template_create_total`
- `template_clone_total`
- `template_ai_generate_total`
- `template_ai_generate_failed_total`
- `template_permission_denied_total`

### 5.3 审计与追踪

- 管理后台的模板、分类、标签变更写入 `TemplateAuditEvent`。
- API 层为每次请求生成或透传 `request_id`，错误返回中必须回传。
- MVP 可先采用应用日志 + 审计表的方式追踪，不强制引入完整分布式 tracing。

## 6. 部署说明

- 继续沿用现有单体 FastAPI 应用部署方式，通过 `uvicorn apps.backend.main:app --host 0.0.0.0 --port 8000` 启动。
- 模板能力作为现有应用的新模块和新路由部署，不新增独立服务进程。
- 部署前需要完成模板相关数据表迁移，包括：
  - `templates`
  - `template_categories`
  - `template_tags`
  - `template_tag_relations`
  - `template_generation_records`
  - `template_audit_events`
- 首次部署需要初始化基础分类、标签和管理员可用的系统模板状态字典。
- AI 生成依赖的模型配置、超时、限流参数通过环境变量注入，禁止写死在代码中。
- 推荐发布顺序：数据库迁移 -> 初始化分类标签与必要配置 -> 部署应用 -> 验证鉴权、模板列表、后台发布和 AI 生成链路。

## 7. 风险与技术债

- PRD 尚未完全确认“模板正文内容”是纯文本、富文本还是结构化 schema；首版若按纯文本落地，后续升级为结构化模板时可能需要数据迁移。
- 当前“使用模板”只定义为向现有创建入口提供起始内容，不同业务入口若需要不同载荷格式，后续可能需要增加适配层。
- 分类与标签若允许后台自由维护，长期可能出现命名重复、粒度失衡和检索质量下降，后续可能需要治理规则或合并机制。
- AI 生成首版为单轮同步调用，易实现但受模型响应时间和输出稳定性影响较大；后续可能需要异步任务、模板化提示词和结果质检。
- 系统模板与用户模板共用主表可以降低 MVP 实现成本，但当后续字段差异明显时，领域模型可能出现膨胀。
- 当前仅设计基础 RBAC 和资源归属控制，不覆盖团队共享、组织空间、多级审批等复杂权限需求。
