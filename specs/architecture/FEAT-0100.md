# Architecture - FEAT-0100

## 总体架构
FEAT-0100 采用单体后端 + 阶段式 AI 工作流的架构。首期不引入独立工作流编排平台，也不将多个 Agent 暴露为分布式微服务，而是在 FastAPI 应用内部以可配置的流程阶段实现“故事编辑 -> 用户确认 -> 写手生成 -> 读者评审 -> 条件回流”的闭环。

系统分为四层：
- 表现层：前台创作端与后台管理端，通过 HTTP API 调用后端服务。
- 应用层：FastAPI 路由、鉴权、配额校验、流程编排、错误返回。
- 领域层：项目管理、章节管理、大纲协作、生成工作流、评审回流、模板管理、会员权益管理。
- 基础设施层：关系型数据库、对象化 AI Provider 适配器、日志与审计、可选缓存。

首期关键设计原则：
- 用户始终保留最终决策权，默认生成链路必须停在人工确认节点。
- 长篇一致性依赖结构化记忆，不依赖单次超长上下文。
- 多 Agent 通过统一的工作流接口抽象，便于后续扩展配图、导出、社区协作等能力。
- 所有权益、阈值、回流上限、模板元数据均配置化，不在业务流程中硬编码。

## 模块划分
### 1. API Gateway / FastAPI Router
- 对外提供 REST API。
- 统一处理认证、参数校验、错误码映射、请求日志。
- 按前台创作域与后台管理域拆分路由。

### 2. Auth & Membership 模块
- 管理用户身份、会员档位、剩余额度、免费次数。
- 在生成、重写、扩写、自动生成启动前执行额度校验。
- 为后续支付系统预留权益快照接口，但首期不接入支付。

### 3. Template 模块
- 管理项目初始化模板。
- 模板内容包括题材建议、篇幅约束、初始世界观骨架、章节结构建议、提示词参数。
- 支持系统模板与用户自定义模板的统一读取模型。

### 4. Project & Outline 模块
- 管理小说项目、章节清单、章节大纲、项目设置。
- 提供大纲编辑能力与章节级定稿状态管理。
- 维护“自动生成是否开启、当前推进到哪一章、是否暂停”状态。

### 5. Memory 模块
- 维护结构化记忆，至少包括角色卡、世界观卡、关键事件、章节摘要。
- 对生成、评审、重写、扩写统一提供上下文组装。
- 在章节确认或重写完成后更新记忆快照。

### 6. Agent Workflow 模块
- 统一封装故事编辑 Agent、写手 Agent、资深读者 Agent。
- 用阶段状态机驱动章节流转。
- 负责有限回流、阈值判断、自动生成推进与人工接管。

### 7. Chapter Editing 模块
- 管理章节正文、版本、段落级重写/扩写请求。
- 保留章节版本历史，避免覆盖式写入导致内容丢失。
- 与 Memory 模块联动，在确认新版本后更新摘要。

### 8. Admin 模块
- 管理模板、会员档位、权益参数、阈值配置。
- 提供启停状态控制与审计日志记录。

### 9. Logging & Audit 模块
- 对核心流程记录结构化日志。
- 对项目创建、章节生成、评分回流、后台配置变更保留审计记录。
- 不记录敏感密钥，不落盘完整提示词明文中的敏感字段。

## API 设计
API 采用 REST 风格，统一返回 JSON：

```json
{
  "request_id": "uuid",
  "data": {},
  "error": null
}
```

错误时：

```json
{
  "request_id": "uuid",
  "data": null,
  "error": {
    "code": "QUOTA_EXCEEDED",
    "message": "monthly quota exceeded"
  }
}
```

核心接口如下。

### 1. 项目与模板
- `GET /api/v1/templates`
  - 查询可用模板列表，支持系统模板和用户模板。
- `POST /api/v1/projects`
  - 创建项目。
  - 入参：`title`、`genre`、`length_type`、`template_id`、`auto_mode_enabled`。
- `GET /api/v1/projects/{project_id}`
  - 查询项目详情、当前流程状态、配额摘要。
- `GET /api/v1/projects/{project_id}/chapters`
  - 获取章节列表与状态。

### 2. 大纲与故事编辑协作
- `GET /api/v1/projects/{project_id}/outline`
  - 获取项目级大纲与章节大纲。
- `PUT /api/v1/projects/{project_id}/outline`
  - 更新项目级或章节级大纲。
- `POST /api/v1/chapters/{chapter_id}/story-options:generate`
  - 调用故事编辑 Agent 生成多个走向/概要候选。
- `POST /api/v1/chapters/{chapter_id}/story-options/{option_id}:confirm`
  - 选择候选方案并定稿。
- `POST /api/v1/chapters/{chapter_id}/story-chat`
  - 与故事编辑 Agent 进行对话式迭代，返回回复与可选更新建议。

### 3. 章节生成与评审
- `POST /api/v1/chapters/{chapter_id}:generate-draft`
  - 基于已确认大纲、结构化记忆生成正文草稿。
- `POST /api/v1/chapters/{chapter_id}:review`
  - 调用资深读者 Agent 评分。
- `POST /api/v1/chapters/{chapter_id}:confirm`
  - 用户确认章节版本并触发记忆更新。
- `POST /api/v1/chapters/{chapter_id}:rewrite`
  - 整章重写。
- `POST /api/v1/chapters/{chapter_id}/paragraphs/{paragraph_id}:rewrite`
  - 段落重写。
- `POST /api/v1/chapters/{chapter_id}/paragraphs/{paragraph_id}:expand`
  - 段落扩写。

### 4. 自动生成控制
- `POST /api/v1/projects/{project_id}:auto-run`
  - 启动或继续自动生成模式。
- `POST /api/v1/projects/{project_id}:pause`
  - 暂停自动生成。
- `GET /api/v1/projects/{project_id}/workflow-status`
  - 查询自动推进进度、剩余回流次数、阻塞原因。

### 5. 后台管理
- `GET /admin/v1/templates`
- `POST /admin/v1/templates`
- `PUT /admin/v1/templates/{template_id}`
- `POST /admin/v1/templates/{template_id}:toggle`
- `GET /admin/v1/membership-plans`
- `POST /admin/v1/membership-plans`
- `PUT /admin/v1/membership-plans/{plan_id}`

API 约束：
- 所有生成类接口必须校验会员权益与剩余额度。
- 章节正文生成前必须验证章节存在已确认大纲。
- 自动回流达到上限后，接口返回阻塞状态，不继续自动重试。

## 存储方案
首期采用关系型数据库作为主存储，推荐 PostgreSQL。

存储分层如下：
- 业务主数据：PostgreSQL，存储项目、章节、模板、会员、版本、评审结果、记忆记录。
- 可选缓存：Redis，用于短时保存会话态、幂等键、热点模板、自动流程锁。若首期资源有限，可先不启用，通过数据库事务 + 状态字段保证一致性。
- 对象存储：首期非必需。由于 MVP 仅处理文本，可直接存数据库；若后续引入配图、导出附件、长日志归档，再增加对象存储。

存储设计原则：
- 章节正文与记忆快照分表存储，避免单表混合导致读写复杂。
- 所有重写/扩写均以新版本落库，不覆盖历史版本。
- 模板、会员权益、回流阈值均数据化配置，避免写死在代码中。
- 对自动生成流程使用行级锁或乐观锁，避免重复推进同一章节。

## 权限与安全
权限模型首期采用两类角色：
- 创作者用户：仅可访问自己的项目、章节、模板与配额数据。
- 管理员：可访问后台模板管理、会员权益配置与审计日志。

安全要求：
- API 层必须做身份认证，推荐 Bearer Token 或 Session 方案；密钥与数据库凭证通过环境变量或密钥管理服务注入，不允许硬编码。
- 资源访问必须做 owner 校验，避免越权访问其他用户项目。
- 后台管理接口必须与前台接口分域或分前缀隔离，并要求管理员角色。
- 对生成类接口加入幂等控制，避免重复点击导致重复扣额和重复生成。
- 对 AI Provider 调用设置超时、重试上限和脱敏日志。
- 对用户输入的大纲、正文、模板内容做长度限制和基础内容清洗，降低提示词注入与异常负载风险。
- 审计日志记录后台配置变更、额度变更和关键内容流转操作。

## 错误处理
系统必须返回明确错误，不允许 silent fail。错误分为以下几类：

### 1. 请求类错误
- `400 BAD_REQUEST`
- 场景：参数缺失、字段格式错误、章节未定稿大纲却请求生成正文。

### 2. 认证与权限错误
- `401 UNAUTHORIZED`
- `403 FORBIDDEN`
- 场景：未登录、越权访问、非管理员调用后台接口。

### 3. 业务状态错误
- `409 CONFLICT`
- 场景：章节当前状态不允许执行该动作、自动流程已暂停、同一章节正在处理中。

### 4. 配额错误
- `402 QUOTA_EXCEEDED` 或 `429 TOO_MANY_REQUESTS`
- 场景：免费次数耗尽、月额度不足、自动生成达到并发限制。

### 5. 外部服务错误
- `502 PROVIDER_ERROR`
- `504 PROVIDER_TIMEOUT`
- 场景：AI Provider 调用失败或超时。

### 6. 系统内部错误
- `500 INTERNAL_SERVER_ERROR`
- 场景：未预期异常、数据库事务失败。

错误处理策略：
- 所有接口返回统一错误结构，包含 `request_id` 便于日志追踪。
- 工作流执行失败时保留当前阶段上下文与失败原因，便于用户重试和人工接管。
- 自动回流遇到连续 Provider 失败不计入内容质量回流次数，但要记录失败次数并停止无限重试。
- 当评审分数低于阈值且回流次数达到上限时，章节状态标记为 `max_retry_reached`，由用户决定下一步。

## 可扩展性说明
当前架构刻意保持单体化，但通过模块边界为扩展预留空间：

### 1. Agent 扩展
故事编辑、写手、读者评审都通过统一 Agent Provider 接口接入。后续可新增配图 Agent、封面 Agent、导出 Agent，而不改变项目和章节主模型。

### 2. 工作流扩展
阶段式状态机可从当前固定链路扩展为更多可配置节点，例如“敏感内容检测”“风格统一化处理”“导出前校验”。首期先将节点配置保留在服务端，不建设可视化编排界面。

### 3. 存储扩展
结构化记忆已独立建模。未来如需引入向量检索或混合检索，可在 `Memory` 模块旁增加检索层，而不破坏现有关系模型。

### 4. 协作能力扩展
当前默认单用户项目制。后续若支持工作室协作，可在 `NovelProject` 外增加 `ProjectMember` 与角色表，并复用现有资源 owner 校验逻辑扩展为 ACL 校验。

### 5. 运营能力扩展
模板和会员权益采用后台配置模型，未来可接支付、订单、活动赠送、数据看板，不需要重写创作流程主链路。

### 6. 性能与可靠性扩展
当生成请求量上升时，可将 Agent Workflow 执行从同步请求切换为异步任务队列，API 层仅负责下发任务与查询状态。该演进不会改变上层 API 语义，只改变执行方式。
