# Decision - FEAT-0001

## 最终选择方案
选择“Web 端小说创作工作台 + 管理后台”的双端 MVP 方案，并采用“章节级多 Agent 协作流水线 + 人工最终确认”的主流程。

创作端以单个小说项目为核心对象，支持用户创建项目、选择题材/篇幅/模板、编辑全书与章节大纲、生成章节正文、查看质量评分，并对章节或段落进行局部重写与扩写。生成链路固定为：故事编辑 Agent 先产出多个章节方向方案，用户选择或继续对话修订后，再由写手 Agent 生成正文，最后由资深读者 Agent 进行评分与点评；评分低于阈值时，系统仅做有限次数自动回流，最终仍由用户确认是否采纳。

后台首期只保留两类最小运营能力：模板管理与会员权益配置。模板支持系统模板和用户私有模板；会员仅管理免费次数、月度额度和额度消耗记录，不引入复杂计费引擎。

## 选择理由
- 与需求卡片中的核心价值一致，优先解决“情节连贯性、可控生成、人工拍板”这三个最关键问题。
- 将生产单元收敛到“章节”，可以同时支撑短篇和长篇的渐进式实现，避免首期直接做全书自动生成导致范围失控。
- 先做固定角色职责的多 Agent 流程，比开放式可编排工作流更容易落地、验证与运营。
- 将连贯性实现限定为“结构化记忆 + 已确认内容摘要”比无限拼接全文更稳定，也更符合长上下文成本控制。
- 后台仅实现模板和额度配置，能满足运营与商业化起步需求，同时避免首期被复杂支付、结算和模板市场拖慢。
- 模板先限制为“系统模板 + 用户私有模板”，可以降低审核、版权和分享机制的不确定性。

## MVP 范围
- 小说项目创建：
  - 支持题材、篇幅、模板选择。
  - 默认一个项目对应一部小说。
- 模板能力：
  - 查看系统模板。
  - 创建、编辑用户私有模板。
  - 模板字段至少覆盖题材设定、角色设定、世界观、文风要求、章节生成要素。
- 大纲能力：
  - 维护全书大纲与章节大纲。
  - 在章节大纲编辑过程中与故事编辑 Agent 多轮对话。
  - 由用户确认章节方向后进入正文生成。
- 章节生成能力：
  - 故事编辑 Agent 生成多个候选方案。
  - 写手 Agent 基于已确认大纲和记忆生成正文。
  - 资深读者 Agent 给出评分与评语。
  - 低分章节触发有限次数自动回流。
- 用户控制能力：
  - 章节内容需人工最终确认后才进入正式章节列表。
  - 提供“自动生成”模式，但范围先限定为单章节自动执行完整流水线，不支持连续全书无人值守生成。
- 局部编辑能力：
  - 对任意段落发起重写或扩写。
- 连贯性机制：
  - 维护角色设定、世界观设定、章节摘要、伏笔记录等结构化记忆。
  - 生成时基于已确认内容和记忆摘要提供上下文。
- 管理后台：
  - 模板管理。
  - 会员权益配置。
  - 免费次数、月度额度、消耗记录管理。

## Out of scope
- AI 配图生成。
- 社区协作、评论、共创、多用户实时协同编辑。
- 模板市场、模板分享、模板审核生态。
- 复杂计费与支付结算体系，例如按字数、模型成本、套餐叠加精细结算。
- 高级数据看板和经营分析。
- EPUB、PDF、Word 等多格式导出。
- 原生移动端 App。
- 全书级连续自动生成。
- 深度内容安全治理、复杂风控与合规审核系统。

## 技术约束
- 后端必须使用 FastAPI。
- 测试必须使用 pytest，不使用 unittest 作为主测试框架。
- 不允许硬编码 secrets、token、数据库密码。
- 设计阶段文档只能写入 `specs/` 允许的路径，本次仅写入 `specs/decision/FEAT-0001.md`。
- 采用最小变更原则，首期不引入替代目录结构或额外设计文档路径。
- 长上下文连贯性优先采用“结构化记忆 + 摘要”策略，不依赖无限原文拼接。
- 核心生成流程需要明确错误返回与日志，避免 silent fail。

## 风险与注意事项
- “情节连贯性”缺少量化验收标准，后续必须在 PRD 或测试计划中补充评分维度与通过阈值。
- 资深读者 Agent 的评分规则、及格线、回流次数若不尽快固定，会影响流程设计、测试设计和会员额度消耗规则。
- “自动生成”若后续扩展到多章节连续执行，必须补充额度保护、失败兜底和人工中断机制。
- 段落级重写/扩写是否保留版本历史尚未明确，若不定义可能影响可追溯性和用户体验。
- 用户私有模板目前不涉及分享和审核，后续若开放社区能力，需要重新补充权限、版权与内容安全策略。
- 长篇小说场景下，结构化记忆的更新策略和摘要质量将直接影响生成稳定性，需要在架构阶段重点设计。
