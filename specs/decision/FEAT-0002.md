# FEAT-0002 决策结论

## 1. 最终选择的方案

- 方案 ID：`solution_2`
- 方案名称：成长积分型会员体系

### 选择理由

本次选择基于 `specs/scoring/FEAT-0002.json` 的评分结果，以及 `specs/brainstorm/FEAT-0002.md` 中对各方案业务目标与实现路径的描述，不采用主观拍板。

- `solution_2` 总分 `45`，为五个方案中最高，高于 `solution_1` 的 `43`、`solution_3` 和 `solution_4` 的 `37`，以及 `solution_5` 的 `32`。
- 它在 MVP 最关键的早期验证维度上最均衡：
  - `feasibility=8`：实现可行性高，不需要首期就建设复杂支付、企业计费或多 SKU 权益编排。
  - `speed=8`：上线速度最快，适合先验证会员体系是否能带动活跃、留存和回访。
  - `cost=7`：建设成本相对可控，能在较小范围内完成闭环。
  - `risk=7`：总体风险低于支付驱动、社群冷启动和企业化重交付方案。
- 从 brainstorm 看，当前输入只有“添加一个会员系统”，没有明确要求必须先做商业化收费。因此更合理的首版路径，是先用低门槛成长机制建立会员状态、权益触发和运营闭环，再决定是否叠加付费层。
- 与 `solution_1` 相比，`solution_2` 放弃了“直接收入验证”的优势，换来更低的接入门槛和更轻的首版复杂度，更适合作为当前项目的 MVP 起点。
- 与 `solution_3`、`solution_5` 相比，`solution_2` 避免过早进入复杂商品化和 B 端组织能力建设，能把首版重心放在用户成长反馈、基础权益管理和运营可配置能力上。

结论：`solution_2` 是当前阶段综合得分最高、最适合做 MVP 的会员系统方案。后续若验证出明确付费意愿，可在其基础上逐步演进到 `solution_1` 的订阅模型，形成“成长 + 付费”的混合会员体系。

## 2. 方案排名（Top 3）

### Top 1: `solution_2` 成长积分型会员体系

- 排名第一，因为它总分最高（`45`），且在可行性、速度、成本、风险四个维度最均衡。
- 它最适合优先验证的核心命题是：会员体系能否先通过签到、任务、积分、等级权益提升留存和活跃，再为后续商业化打基础。
- 它对当前需求最稳妥，因为不依赖项目已经具备强支付转化、强社区属性或团队协作场景。

### Top 2: `solution_1` 订阅付费型会员体系

- 排名第二，因为它总分 `43`，与第一名差距很小，且商业路径最直接。
- 它在 `market=8`、`scalability=8` 上表现强，说明长期收入模型和权限扩展能力都较好。
- 但 `cost=6`、`risk=6` 反映出支付接入、退款对账、续费和合规会明显抬高首版复杂度，因此不优先作为 MVP 起点。

### Top 3: `solution_3` 权益通行证型会员体系

- 排名第三，因为它与 `solution_4` 同为 `37` 分，但其 `market=8`、`scalability=9` 明显高于社群方案，更适合作为后续商业化扩展方向。
- 它的优势是权益商品化灵活，适合未来做按模块、按场景售卖。
- 但 `feasibility=6`、`cost=4`、`speed=5` 表明首版建设和运营复杂度偏高，不适合当前阶段直接落地。

## 3. 放弃其他方案的原因

### 放弃 `solution_1` 的原因

- trade-off：放弃“更直接的收入验证”，换取“更快上线、更低实现复杂度和更低初期门槛”。
- 该方案虽然商业模式清晰，但首版必须同步处理支付集成、订单账单、续费提醒、退款对账和权益到期控制。
- 如果当前项目价值感和转化链路尚未验证，过早引入付费墙会增加增长风险，也会让 MVP 成本偏高。

### 放弃 `solution_3` 的原因

- trade-off：放弃“更灵活的权益商品化售卖能力”，换取“更简单、用户更易理解的单一成长体系”。
- 该方案需要处理通行证 SKU、权益绑定、叠加规则、时效控制和使用次数控制，前后端授权判断更复杂。
- 对当前“先把会员系统建立起来”的目标来说，它过重，容易把 MVP 做成商品编排系统。

### 放弃 `solution_4` 的原因

- trade-off：放弃“社群归属感和互动壁垒”，换取“避免社区冷启动和持续运营重投入”。
- 该方案成立的前提是项目本身有明确社区属性，并且团队能持续投入内容运营、活动组织和审核风控。
- 当前需求没有体现强社群场景，因此先做社群身份会员，容易出现价值感不足的问题。

### 放弃 `solution_5` 的原因

- trade-off：放弃“高客单价的企业收入潜力”，换取“保留首版轻量、面向通用用户的落地路径”。
- 企业席位方案要求组织模型、席位分配、企业计费、审计日志、合同发票等重能力。
- 如果当前项目仍处于基础会员能力建设阶段，这条路线会让 MVP 偏离目标，交付周期和风险都过高。

## 4. MVP 范围

以下内容应直接进入后续 PRD，作为首版必须交付的功能范围：

- 会员基础身份：用户拥有明确的会员等级字段，至少支持 `普通用户`、`成长会员` 两类状态。
- 积分账户：每个用户有独立积分余额、累计获得积分和消耗记录。
- 成长任务中心：支持配置并展示基础任务，如每日签到、完成指定行为、连续活跃。
- 积分发放规则：系统可根据任务完成结果自动发放积分，并记录来源、时间和数量。
- 会员等级规则：基于累计积分或成长值自动升级会员等级，支持查看当前等级和下一等级门槛。
- 等级权益映射：不同等级可解锁不同权益，如专属标识、功能额度提升、优惠券资格或优先体验资格。
- 会员主页/面板：用户可查看当前等级、积分余额、成长进度、已解锁权益和待完成任务。
- 积分流水记录：用户可查询积分获得与消耗历史，便于解释成长状态变化。
- 基础奖励发放：支持至少一种可落地奖励形式，例如优惠券、权益标识或功能开关解锁。
- 后台规则配置：运营可配置任务开关、积分值、等级门槛和基础权益映射。
- 基础风控限制：对签到、重复触发任务、异常高频获取积分做最小必要限制，避免明显刷分。
- 事件与日志记录：核心会员动作需保留事件记录，便于后续分析留存、任务完成率和等级分布。

## 5. Out of Scope

本轮明确不做以下内容：

- 不做支付、订阅扣费、自动续费、退款对账和发票能力。
- 不做复杂积分商城，不支持多类商品兑换和大规模库存管理。
- 不做多层级、多条件、可编排的复杂规则引擎，首版仅支持有限的固定任务类型。
- 不做完整反作弊系统，不做设备指纹、复杂风控模型或黑产识别。
- 不做社群专区、会员论坛、直播活动、圈层互动和内容共创能力。
- 不做企业组织、席位管理、团队权限、统一结算和 SSO。
- 不做多 SKU 通行证、权益包叠加、按次数购买或按场景售卖体系。
- 不做跨渠道营销自动化、精细化 CRM 分群和复杂数据看板。

## 6. 推荐技术约束

- Backend：`FastAPI`
- Tests：`pytest`
- 数据建模：会员等级、积分账户、任务配置、积分流水、权益映射保持最小必要模型
- 架构原则：先采用清晰的规则配置 + 事件记录模式，避免首版引入重型规则引擎
- 风控原则：仅做最小必要防刷限制，不为首版引入复杂风控平台
- 配置原则：任务、积分值、等级门槛、权益映射必须可配置，禁止硬编码业务参数
- 安全约束：不得硬编码 secrets、token、数据库密码
- 日志要求：会员升级、积分发放、奖励发放等核心流程必须有日志

## 决策总结

`solution_2` 不是最终商业形态，而是当前最合适的第一阶段方案。它以最高评分证明：在需求尚未明确指向强付费或强 B 端场景时，先建立成长型会员闭环，能以更低成本验证用户活跃、留存与权益感知；待机制跑通后，再叠加订阅付费能力，会比一开始直接做重商业化体系更稳妥。
