# Decision - FEAT-0100

## 最终选择方案
首期采用“单用户项目制 + 多阶段 AI 工作流 + 人工审批优先”的方案。

产品主链路确定为：
1. 用户创建小说项目，选择题材、篇幅和模板。
2. 用户在项目级大纲编辑器中查看和编辑章节大纲，并与故事编辑 Agent 进行多轮讨论。
3. 故事编辑 Agent 为目标章节生成多个走向方案和概要，用户必须先做选择或修改后定稿。
4. 写手 Agent 基于已确认大纲、结构化记忆和既有章节摘要生成正文。
5. 资深读者 Agent 对生成结果按预设维度评分。
6. 若评分低于阈值，则在有限次数内自动回流至前序生成环节；若达到上限，则交由用户决策。
7. 用户可对章节整体进行确认或要求重写，也可对段落执行扩写和重写。

技术实现上，首期不把多 Agent 暴露为复杂编排平台，而是落为后端可配置的阶段式流程。长篇一致性依赖结构化记忆，而不是完全依赖单次超长上下文。

## 选择理由
该方案与需求卡片中的核心价值一致，即“AI 协作生成”与“用户保留最高决策权”并存。默认人工确认能够降低长篇创作中的失控风险，自动生成模式则作为效率增强选项保留。

将多 Agent 收敛为阶段式工作流，能够显著降低首期架构复杂度，同时保留后续扩展到更细粒度编排的空间。对首期而言，用户感知的是稳定的创作体验，不需要先建设复杂的工作流可视化系统。

长上下文问题采用“角色卡、世界观卡、关键事件摘要、章节摘要”等结构化记忆方案更稳妥。这比完全依赖一次性长上下文生成更可控，也更适合后续做章节级回流、重写和运营配额控制。

后台范围限定为模板管理和会员权益配置，可以覆盖需求中最明确的运营诉求，同时避免把首期资源分散到复杂支付、财务或 BI 能力。

## MVP 范围
- 小说项目创建，字段至少包括题材、篇幅、模板选择。
- 项目级大纲编辑器，支持章节大纲查看、编辑和与故事编辑 Agent 的对话式迭代。
- 故事编辑 Agent 为章节生成多个备选走向和概要，由用户确认。
- 写手 Agent 基于已确认大纲和结构化记忆生成章节正文。
- 资深读者 Agent 对章节进行评分，并在低于阈值时触发有限次数自动回流。
- 用户对章节执行确认、整章重写、段落扩写、段落重写。
- 项目级结构化记忆，至少包含角色信息、世界观设定、关键事件和章节摘要。
- 管理后台最小能力：模板管理、会员档位配置、免费次数和月额度配置。
- 自动生成模式作为可选项提供，但必须支持用户随时暂停和接管。

## Out of scope
- AI 配图生成。
- 社区协作、评论互动、作品分享。
- 多人实时协作编辑和团队权限体系。
- 数据看板、复杂运营分析和 BI。
- 电子书、PDF、Word 等多格式导出链路。
- 支付、结算、发票和财务系统集成。
- 可视化工作流编排平台。
- 面向多语言市场的能力。

## 技术约束
- 后端技术栈必须使用 FastAPI。
- 测试框架必须使用 pytest，不使用 unittest 作为主测试框架。
- 不允许硬编码 secrets、token 或数据库密码。
- 首期应保持最小变更和最小实现闭环，不为“未来可能需求”提前引入复杂基础设施。
- 评分与回流机制需要明确最大回流次数和失败兜底策略，避免无限循环。
- 结构化记忆必须可被生成、评分和重写流程复用，避免上下游各自维护割裂状态。
- 模板与会员配置应采用可管理的后台配置模型，不在代码中硬编码权益规则。

## 风险与注意事项
- 需求中的“多 Agent”如果过早实现为真实独立编排节点，会显著增加开发复杂度和可观测性成本，首期应坚持阶段式工作流抽象。
- 章节评分标准尚未明确，若没有统一维度和阈值，自动回流容易产生不稳定结果。建议至少固定连贯性、文风一致性、节奏和角色行为合理性四类评分维度。
- 自动生成模式存在成本失控和方向漂移风险，必须设置中止条件、消耗提示和人工随时接管能力。
- 模板定义粒度目前不清晰，首期应先限定为“项目初始化模板”，避免同时支持题材模板、文风模板、章节模板等多种异构模板。
- 会员权益边界尚未完整定义，首期建议只控制免费次数、月额度和基础生成配额，不引入模型分级和复杂并发策略。
- 长篇一致性高度依赖记忆摘要质量，因此摘要更新机制必须纳入主流程，而不是作为附属功能补做。
- 当前需求未明确多人协作与权限模型，首期决策默认单用户项目制；若后续转向工作室场景，数据模型和后台权限会受到影响。
- 本次仅根据任务要求写入决策文档；若后续需要 `specs/intake/FEAT-0100.json`，应在明确 intake schema 后单独生成。
