# FEAT-0002 PRD

## 1. 业务背景

当前项目提出了“添加一个会员系统”的需求，但现阶段没有明确要求优先建设付费订阅、企业席位或复杂权益商品化能力。结合现有决策材料，本期应优先落地“成长积分型会员体系”，以较低门槛建立用户成长反馈、等级变化、权益触发和基础运营闭环。

本期 MVP 的核心业务目标不是直接验证收入，而是验证会员体系是否能够提升用户活跃、留存和回访频率，并为后续叠加付费会员、权益包或更复杂运营策略提供基础能力。

## 2. 用户角色与场景

### 用户角色

- 普通用户：已注册并使用当前产品，希望通过日常行为获得成长反馈和会员权益。
- 成长会员：已通过积分和成长值达到更高等级的用户，关注等级进度、已解锁权益和奖励。
- 运营人员：负责配置任务、积分值、等级门槛和权益映射，观察会员机制是否有效。

### 典型场景

- 用户每日访问产品并完成签到，获得积分，看到成长进度被更新。
- 用户完成指定行为后自动获得积分，并在满足门槛时升级为更高会员等级。
- 用户进入会员面板查看当前等级、积分余额、成长进度、权益内容和任务完成情况。
- 用户查看积分流水，理解积分增加或消耗的来源，降低对规则不透明的疑虑。
- 运营人员调整任务开关、积分值、等级门槛和基础权益映射，以支持 MVP 运营实验。

## 3. 用户故事

- 作为普通用户，我希望通过签到和完成任务获得积分，这样我能感受到持续使用产品有明确回报。
- 作为普通用户，我希望看到当前等级、距离下一等级还差多少成长值，这样我知道下一步该做什么。
- 作为成长会员，我希望在升级后立即看到可用权益，这样我能明确感知会员体系带来的价值。
- 作为任意会员用户，我希望能查看积分流水和奖励记录，这样我能理解我的会员状态为什么发生变化。
- 作为运营人员，我希望配置任务、积分值、等级门槛和权益映射，这样我可以在不改代码的情况下调整会员策略。
- 作为平台管理员，我希望系统对重复签到和异常高频获取积分做基础限制，这样可以避免明显刷分破坏规则公平性。

## 4. In Scope / Out of Scope

### In Scope

- 会员基础身份能力，至少支持 `普通用户` 和 `成长会员` 两类状态，并支持等级展示。
- 积分账户能力，包括积分余额、累计获得积分和基础消耗记录。
- 成长任务中心，首版支持有限的固定任务类型，如每日签到、完成指定行为、连续活跃。
- 积分发放规则，根据任务完成结果自动发放积分，并记录来源、时间和数量。
- 会员等级规则，基于累计积分或成长值自动升级，并展示当前等级与下一等级门槛。
- 等级权益映射，不同等级可解锁不同权益，如专属标识、功能额度提升、优惠券资格或优先体验资格。
- 会员主页或面板，展示当前等级、积分余额、成长进度、已解锁权益和待完成任务。
- 积分流水查询，支持用户查看积分获得与消耗历史。
- 基础奖励发放，首版至少支持一种实际奖励形式落地。
- 后台规则配置，支持运营配置任务开关、积分值、等级门槛和基础权益映射。
- 基础风控与日志，对签到、重复触发任务、异常高频获取积分做最小必要限制，并记录核心会员事件。

### Out of Scope

- 支付、订阅扣费、自动续费、退款对账、发票和其他收费能力。
- 复杂积分商城、多类商品兑换、大规模库存管理。
- 多层级、多条件、可编排的复杂规则引擎。
- 完整反作弊平台、设备指纹、复杂风控模型或黑产识别。
- 社群专区、会员论坛、直播活动、圈层互动和内容共创能力。
- 企业组织、席位管理、团队权限、统一结算和 SSO。
- 多 SKU 通行证、权益包叠加、按次数购买和按场景售卖能力。
- 复杂 CRM 分群、营销自动化和高级运营数据看板。

## 5. 验收标准

- 用户登录后具备明确会员状态，系统能区分 `普通用户` 与 `成长会员`，并正确展示当前等级。
- 每个用户都有独立积分账户，可查看积分余额、累计获得积分和积分流水。
- 用户完成首版支持的任务后，系统可自动发放积分，并记录任务来源、发放时间和积分数值。
- 当用户达到预设成长门槛时，系统可自动完成会员升级，并在会员面板展示升级结果。
- 用户可在会员面板查看当前等级、成长进度、下一等级门槛、已解锁权益和待完成任务。
- 不同会员等级对应的权益映射可生效，首版至少有一种权益能真实触发并被用户感知。
- 运营人员可通过后台配置任务开关、积分值、等级门槛和基础权益映射，配置生效不依赖代码修改。
- 系统对重复签到和异常高频获取积分具备基础限制，明显重复触发不能无限累积积分。
- 会员升级、积分发放、奖励发放等核心动作有事件记录和日志，支持后续追溯和基础分析。
- 功能实现需符合项目工程约束：后端基于 FastAPI，测试基于 pytest，不硬编码业务敏感配置。

## 6. 风险与待确认项

### 风险

- 当前原始需求较泛，仅说明“添加会员系统”，如果后续业务方实际期望是付费会员，当前 MVP 可能无法直接满足收入目标。
- 奖励价值若不够明确，用户可能无法感知成长会员的价值，导致活跃提升有限。
- 任务规则和等级门槛若设计不合理，可能造成升级过快、升级过慢或用户对公平性的质疑。
- 基础风控若过弱，可能出现刷分行为，影响会员等级与奖励机制可信度。
- 现有项目若缺少明确的“可解锁权益”，会员体系可能只剩积分展示，难以形成闭环。

### 待确认项

- 首版“指定行为”具体包含哪些业务动作，哪些动作应计入成长任务。
- 首版会员等级是否仅需两档，还是需要扩展为多档成长等级。
- 首版最小可落地权益是什么，应优先选择优惠券、功能额度、身份标识还是优先体验资格。
- 积分是否允许消耗；如果允许，首版可消耗场景是什么。
- 会员面板、任务中心和后台配置页是否已有现成入口，还是需要新增页面承载。
- 运营后台的使用角色和配置权限边界如何定义。
