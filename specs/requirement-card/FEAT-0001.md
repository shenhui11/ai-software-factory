# Requirement Card - FEAT-0001

## 原始需求
我计划开发一个名为“智能小说协同生成平台”的系统。其核心目标是构建一个由“多种AI智能体协作+人工审核决策”驱动的创作环境，用于生成情节连贯、可深度定制的小说。在功能上，系统前端允许用户创建项目时选择小说种类（如言情、玄幻、架空、恐怖）、篇幅（长篇/短篇）并选用模板（系统提供或用户自定义）；系统需内置一个强大的后台，用于管理这些模板和会员制度（可配置免费次数、月度额度等权益）。创作流程的关键在于解决“情节连贯性”问题，避免记忆断层。为此，系统设计了多智能体协作流水线：首先，由一组“故事编辑Agent”对后续章节进行脑暴，生成多个情节走向和主题概要方案，供用户选择；确定方向后，由一组有经验的“写手Agent”对章节内容进行丰富、细节补充和风格化润色；最终，由一位“资深读者Agent”对生成章节进行质量评估与打分，若分数低于预设标准，则自动回流至前述环节进行重写或补充，形成质量闭环。在整个过程中，用户始终拥有最高决策权：默认模式下，每个章节的内容需经人工最终确认，但用户可选择点击“自动生成”来交由AI连贯执行；系统提供专门的大纲编辑器，支持用户查看和编辑任何章节的大纲，并且在编辑时能与“故事编辑Agent”进行实时对话讨论，迭代优化方案，最终由用户拍板定案。对于已生成的内容，用户可对任意段落发起“重写”或“扩写”，由AI进行细节再生。此外，系统还需考虑扩展性，未来可支持生成配图、社区协作、数据看板、多格式导出等功能。在技术上，需确保AI生成长上下文记忆管理以保障连贯性，并构建可靠的后台来运营模板和会员服务。

## 当前理解
该需求面向“AI 协作写作”场景，目标是提供一个以小说项目为单位的创作平台。平台核心不只是文本生成，而是把“项目配置、模板管理、章节大纲、章节生成、质量评估、人工确认、会员运营”整合为一个完整系统。

我当前理解的最小主流程如下：
1. 用户创建小说项目，选择题材、篇幅和模板。
2. 用户进入大纲或章节创作阶段，先由故事编辑 Agent 产出多个方向方案。
3. 用户选择一个方向，或者继续与故事编辑 Agent 对话修改大纲。
4. 写手 Agent 基于已确认的大纲和上下文生成章节正文。
5. 资深读者 Agent 对章节进行评分与点评。
6. 若评分未达阈值，则系统触发回流重写；若达标，则交给用户确认。
7. 用户可继续下一章节，或对已有段落执行重写、扩写等局部操作。

系统还包含两类后台能力：
- 内容运营后台：管理模板、模板分类、系统默认模板与用户自定义模板。
- 商业化后台：管理会员套餐、免费次数、月度额度等权益配置。

## 目标用户
- 小说创作者：希望快速生成可控、连贯的小说内容，并保留最终决策权。
- 内容工作室或编辑团队：希望通过模板和协同流程批量化产出小说内容。
- 平台运营人员：需要管理模板、会员权益和内容生产规则。
- 后续可能扩展的高级用户：需要更长篇幅、更细粒度控制、更高额度或团队协作能力。

## 核心业务目标
- 提供可配置的小说项目创建能力，降低创作启动成本。
- 通过多 Agent 流水线提升章节创作效率与质量。
- 通过长上下文记忆和大纲管理提升情节连贯性，减少设定遗失和剧情断层。
- 保持“人工最终拍板”的创作控制权，同时支持自动生成模式提升效率。
- 通过模板体系提升复用性和可定制性。
- 通过会员与额度体系支撑平台商业化运营。

## 需求问题澄清
- “项目”与“小说”的关系是一对多，一个项目就是一个平台，可以承载不同的小说需求。
- 模板的结构边界：至少包括题材、设定说明、角色设定、文风要求、章节生成提示要素。
- 用户自定义模板支持：创建、编辑、查看、删除。
- 默认流程以“章节”为最小生产单元，大纲先行，正文后出。
- 长篇与短篇的业务规则：影响章节数、上下文窗口、生成策略和计费。
- “故事编辑 Agent”与“写手 Agent”是不同职责的固定角色（不同的角色不同的系统提示词）。
- “资深读者 Agent”的评分维度、分值区间、及格线和回流规则尚未定义  -- 给点建议
- 自动回流的最大次数：5次；失败兜底策略：失败之后由人工兜底确认。
- “自动生成”是自动生成单章，单章输出3000中文字符以内/次，对次数进行统计。
- 大纲编辑器中的“实时对话”不要求流式响应，但是需要多轮上下文保留、编辑记录版本化。
- 段落级“重写/扩写”要求保留原文版本、差异对比和局部确认。
- 后台会员体系的计费对象是什么，暂时按章节计费，后续再次优化。
- 免费次数、月度额度的消耗规则和重置规则 --  按照套餐进行编辑。
- 不需要管理员审核用户自定义模板，避免违规内容。
- 需要多端支持， Web 前端、移动端、自适应页面、管理后台独立端。
- 内容安全、敏感题材限制、模型输出审查机制未定义。
- 首期以 Web 端为主，自适应页面，包含创作端和管理后台。
- 默认流程以“章节”为最小生产单元，大纲先行，正文后出。
- 用户默认拥有章节最终确认权；未确认内容不视为正式定稿。
- 自动生成模式默认仍遵守质量评估与回流规则，只是减少人工逐步确认。
- 连贯性依赖“小说全局设定 + 已确认大纲 + 已确认章节摘要/记忆”而非无限原文拼接。
- 会员权益先按“生成次数/额度”管理，复杂计费延后。
- 用户自定义模板首期仅对模板创建者自己可见，不涉及社区共享审核。
- 首期不处理插图生成、社区协作和复杂数据分析，只保留未来扩展空间。

## 建议 MVP
- 提供小说项目创建页，支持选择题材、篇幅、模板。
- 提供基础模板管理能力：系统模板查看、用户自定义模板创建与编辑。
- 提供大纲编辑器：可查看全书大纲、章节大纲，并与故事编辑 Agent 多轮对话后确认章节方向。
- 提供单章节生成流程：
  - 故事编辑 Agent 生成多个章节方案供用户选择。
  - 写手 Agent 基于已选方案生成正文。
  - 资深读者 Agent 打分并给出评语。
  - 低于阈值时自动回流一次或有限次数。
- 提供章节确认机制：用户确认后才进入正式章节列表。
- 提供段落级重写和扩写能力。
- 提供最小可用的连贯性记忆机制：维护角色设定、世界观设定、章节摘要、伏笔记录。
- 提供管理后台中的模板管理和会员权益配置。
- 提供基础会员规则：免费次数、月度额度、额度消耗记录。

## Out of scope
- AI 配图生成。
- 多人社区协作、评论互动、共创工作流。
- 高级数据看板和经营分析报表。
- 多格式导出，如 EPUB、PDF、Word 批量导出。
- 复杂计费系统，如按模型、字数、套餐叠加精细结算。
- 模板市场、模板分享与审核生态。
- 原生移动端 App。
- 深度内容安全治理和复杂风控系统。

## 待确认项
- 首期必须上线的角色端有哪些：创作者端、管理员端、超级管理员端是否都需要。
- 章节生成的验收标准是什么，尤其是“情节连贯性”的可衡量标准。
- 大纲编辑器的优先级是否高于正文编辑器。
- 自动生成模式的边界是什么，是否允许全书连续生成。
- 会员额度的消耗单位与套餐设计是否已有明确商业规则。
- 用户自定义模板是否需要支持导入导出。
- 资深读者 Agent 的评分是否直接影响是否可发布，还是仅作为建议。
- 回流次数、阈值和人工兜底策略是否需要在 MVP 就支持可配置。
- 是否需要保留每次生成的版本历史和操作审计。
- 是否需要在首期支持长篇小说的超长上下文连续创作，还是先聚焦短篇和中篇场景。
