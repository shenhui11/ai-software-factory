# Requirement Card - FEAT-0001

## 原始需求
我计划开发一个名为“智能小说协同生成平台”的系统。其核心目标是构建一个由“多种 AI 智能体协作 + 人工审核决策”驱动的创作环境，用于生成情节连贯、可深度定制的小说。在功能上，系统前端允许用户创建项目时选择小说种类（如言情、玄幻、架空、恐怖）、篇幅（长篇/短篇）并选用模板（系统提供或用户自定义）；系统需内置一个强大的后台，用于管理这些模板和会员制度（可配置免费次数、月度额度等权益）。创作流程的关键在于解决“情节连贯性”问题，避免记忆断层。为此，系统设计了多智能体协作流水线：首先，由一组“故事编辑 Agent”对后续章节进行脑暴，生成多个情节走向和主题概要方案，供用户选择；确定方向后，由一组有经验的“写手 Agent”对章节内容进行丰富、细节补充和风格化润色；最终，由一位“资深读者 Agent”对生成章节进行质量评估与打分，若分数低于预设标准，则自动回流至前述环节进行重写或补充，形成质量闭环。在整个过程中，用户始终拥有最高决策权：默认模式下，每个章节的内容需经人工最终确认，但用户可选择点击“自动生成”来交由 AI 连贯执行；系统提供专门的大纲编辑器，支持用户查看和编辑任何章节的大纲，并且在编辑时能与“故事编辑 Agent”进行实时对话讨论，迭代优化方案，最终由用户拍板定案。对于已生成的内容，用户可对任意段落发起“重写”或“扩写”，由 AI 进行细节再生。此外，系统还需考虑扩展性，未来可支持生成配图、社区协作、数据看板、多格式导出等功能。在技术上，需确保 AI 生成长上下文记忆管理以保障连贯性，并构建可靠的后台来运营模板和会员服务。

## 当前理解
- 这是一个面向小说创作者的 AI 协同创作平台，强调“多 Agent 分工 + 用户最终决策”。
- 平台核心价值不是单次文本生成，而是围绕长篇创作建立可持续的章节规划、上下文记忆、质量回流和人工把关机制。
- 用户在创建项目时需要完成基础设定：题材、篇幅、模板选择；随后围绕项目大纲和章节持续协作。
- 章节生成流程至少包含三类角色：故事编辑 Agent、写手 Agent、资深读者 Agent，且它们之间存在串行协作与质量回流。
- 用户既可以逐章审批，也可以开启较高自动化程度的“自动生成”模式，但系统仍需保留人工可介入能力。
- 平台除了创作前台，还需要一个运营后台来管理模板、会员权益和可能的资源消耗策略。
- 连贯性保障是第一优先级，需要通过长期记忆、章节摘要、角色设定、世界观状态等机制避免情节断层。

## 目标用户
- 个人网文作者或轻小说作者，希望提升创作速度并保持章节连贯。
- 工作室型内容生产团队，需要多人配合但希望先用 AI 降低初稿成本。
- 平台运营人员，需要维护模板、会员套餐、配额和使用策略。
- 审核型创作者，愿意保留人工拍板权，不接受完全黑盒式自动写作。

## 核心业务目标
- 提供一个可持续生成中长篇小说的协同创作工作流，而非一次性文本生成工具。
- 降低小说策划、大纲扩展、章节撰写和润色的时间成本。
- 通过多 Agent 分工和质量评分回流，提高章节质量和情节连贯性。
- 通过模板和会员体系支撑平台商业化与标准化运营。
- 为后续扩展配图、社区、数据看板、多格式导出等能力预留产品空间。

## 不明确问题
- 项目的首发形态是什么：Web 平台、后台管理系统、还是前后台一体的产品？
- 是否需要登录、注册、团队协作、权限分级，还是先仅支持单用户创作？
- 小说项目的最小数据结构是什么：书籍 > 卷 > 章，还是仅书籍 > 章节？
- 模板的定义范围是什么：仅包含题材提示词与大纲框架，还是包含角色设定、世界观、文风、节奏规则等完整配置？
- 用户自定义模板是否需要可视化管理、版本控制、分享/复制能力？
- “故事编辑 Agent”与用户的实时对话，是否要求真正流式实时协作，还是近实时问答即可？
- “写手 Agent”是单个角色还是多种写作风格 Agent 可选？
- “资深读者 Agent”的评分维度有哪些，阈值由谁配置，是否向用户可见？
- 自动回流的最大次数、终止条件、失败兜底策略是什么？
- “自动生成”模式下，是自动推进单章，还是可连续生成多个章节直到用户暂停？
- 长上下文记忆的具体边界是什么：记住全文、记住摘要、还是按角色/设定卡片记忆？
- 段落级“重写”和“扩写”是否要求保留上下文一致性检查与差异对比？
- 会员权益如何计量：按项目数、章节数、字数、调用次数、Token 消耗，还是组合方式？
- 免费次数、月度额度是否按模型、按 Agent、按功能模块分别计算？
- 后台是否需要运营报表、手动调额、模板审核、会员订单管理等完整管理能力？
- 是否存在内容安全、敏感题材过滤、版权声明与数据留存要求？

## 暂定假设
- 首期以 Web 平台为主，包含创作前台和基础后台管理。
- 首期面向单用户创作场景，不处理多人实时协同编辑。
- 项目结构以“小说项目 > 总纲 > 章节大纲 > 章节正文”为核心，不先引入复杂卷册体系。
- 系统模板与用户自定义模板都以“题材 + 世界观/角色设定 + 大纲框架 + 风格约束”组成。
- 创作主流程为：选模板/建项目 -> 生成候选章节方向 -> 用户确认 -> 生成章节正文 -> 质量评分 -> 低分回流 -> 用户确认入库。
- 用户默认逐章审核；自动生成模式只在同一项目内连续推进章节，但仍允许用户随时中断。
- 连贯性先通过结构化记忆实现，包括全书摘要、角色卡、世界观设定、已发生事件摘要和章节状态。
- 会员体系首期只支持基础套餐与配额控制，不先接入复杂计费和财务对账。
- 首期不覆盖配图、社区协作、数据看板、多格式导出等扩展能力，只保留可扩展接口空间。

## 建议 MVP
- 用户可以创建小说项目，并选择题材、篇幅、模板。
- 支持系统模板与用户自定义模板的创建、编辑和使用。
- 提供总纲与章节大纲编辑器，允许用户查看、编辑并与故事编辑 Agent 对话优化方案。
- 支持章节候选方案生成，至少返回多个情节走向/主题概要供用户选择。
- 支持基于已确认大纲生成章节正文，并由写手 Agent 做细节扩写和风格润色。
- 支持资深读者 Agent 对章节进行评分和简要反馈；当分数低于阈值时触发有限次数回流。
- 支持段落级重写和扩写。
- 支持“人工确认模式”和“自动生成模式”两种创作方式。
- 提供最基础的长上下文记忆管理：项目设定、角色卡、章节摘要、已发生事件。
- 提供后台管理模板和会员套餐/额度配置的基础能力。

## Out of scope
- AI 配图生成与图文联动创作。
- 社区发布、评论、点赞、协作共创等社交能力。
- 高级数据看板、经营分析和复杂 BI 报表。
- 多格式导出（PDF、EPUB、DOCX、出版社格式包等）。
- 复杂支付结算、发票、渠道分销和营销活动系统。
- 多团队、多角色审批流和企业级权限系统。
- 完整移动端 App 与离线创作体验。

## 待确认项
- 首期目标用户更偏个人创作者还是内容工作室，这会影响权限和后台设计。
- 模板体系的最小字段集合，需要明确是否包含人物卡、世界观卡和文风约束。
- 会员权益计量方式与业务规则，需要尽早确定以免后续重构额度模型。
- 章节质量评分标准、阈值和回流次数，需要产品和运营共同定义。
- 自动生成模式的推进边界，需要明确是否允许跨多章连续生成。
- 大纲编辑器与 Agent 对话的交互深度，需要确认是否必须支持流式实时协作。
- 连贯性记忆方案的验收标准，需要明确什么情形可判定为“避免记忆断层”。
- 后台范围需确认是“基础配置后台”还是“可直接运营的完整后台”。
