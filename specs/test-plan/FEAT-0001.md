# FEAT-0001 Test Plan

## 1. 测试策略

### 1.1 测试目标

验证 `FEAT-0001` 的 MVP 是否满足“编辑主导的人机共创写作台”的核心闭环：

- 用户可创建并维护故事项目与核心设定。
- 系统可生成故事骨架与单章节初稿。
- 用户可对指定段落执行受控编辑，并保留候选结果而非直接覆盖正文。
- 系统可执行章节 QA 扫描，并支持基于单条问题的定点修复。
- 用户可比较版本差异、执行回退，并导出为 `Markdown` 或纯文本。

### 1.2 测试范围

覆盖 PRD 中全部验收标准，以及 architecture 中定义的核心接口与边界约束：

- 项目管理：创建、读取、更新项目。
- 核心设定：角色卡、世界观摘要、风格约束维护。
- 骨架生成：梗概、核心冲突、角色摘要、章节级简纲。
- 章节草稿：章节创建、初稿生成、内容保存。
- 段落编辑：`rewrite`、`expand`、`compress`、`tone_shift`、`style_shift`。
- QA 扫描：结构化问题列表与问题类型覆盖。
- 定点修复：基于单条 QA 问题执行局部修复。
- 版本能力：快照、候选版本、diff、回退。
- 导出：`md`、`txt`。

### 1.3 不测试项

- 多人协作、权限审批流、章节锁定与合并。
- 复杂角色状态推演、关系图谱和长期时间线模拟。
- 商业化计费、模板市场、私有化部署。
- “完全消除 AI 痕迹”的文本质量承诺。

### 1.4 方法与原则

- 以 API 行为验证为主，确保 FastAPI 接口符合 PRD 和架构定义。
- 以结构化断言为主，不对 LLM 文本内容做脆弱的逐字匹配。
- 重点验证边界约束：
  - `outline` 不直接写章节正文。
  - `edit` 只产生候选内容，不直接覆盖正文。
  - `qa` 只诊断，不修改内容。
  - `revision` 只允许定点修复，不允许默认整章重写。
  - `versioning` 是唯一回退入口。
- 优先覆盖“最小闭环”主路径，再覆盖失败路径和高风险回归点。

## 2. Happy Path

### 2.1 创作闭环主路径

1. 创建故事项目并保存基础参数。
2. 维护核心设定面板，补充角色卡、世界观摘要、风格约束。
3. 生成故事骨架，得到梗概、核心冲突、角色摘要、章节级简纲。
4. 基于简纲创建章节并生成章节初稿。
5. 保存人工编辑后的章节正文。
6. 对选中段落执行一种编辑操作，获得候选改写结果。
7. 发起章节 QA 扫描，获得结构化问题列表。
8. 针对单条 QA 问题执行定点修复，返回修复候选版本和影响范围。
9. 创建版本快照，查看版本 diff，并执行回退。
10. 以 `md` 或 `txt` 导出当前作品内容。

### 2.2 验收重点

- 每一步都能被后续流程引用，形成完整上下文链路。
- 章节内容在编辑、QA、修复、回退后仍可读取且版本关系清晰。
- 核心设定在骨架生成、章节生成、编辑和 QA 过程中保持可引用。

## 3. Edge Cases

- 项目创建缺少必填字段，返回统一错误结构。
- 项目存在，但骨架尚未生成时尝试生成章节初稿。
- 章节存在，但正文为空时执行段落编辑或 QA 扫描。
- `selection_start`、`selection_end` 越界、倒置或命中空片段。
- `operation` 不在允许枚举中。
- QA 扫描返回空问题列表时，系统仍需提供合法响应。
- 基于不存在的 `issue_id` 或与章节不匹配的 `issue_id` 执行修复。
- 定点修复返回的影响范围超出问题片段范围。
- 对不存在的 `version_id` 执行 diff 或回退。
- 使用不支持的导出格式。
- 章节级回退后，当前活动版本必须更新，旧版本仍保留。
- 角色卡或风格约束更新后，后续生成结果应引用最新设定，而非缓存旧值。

## 4. Permission Cases

说明：MVP 为单人工作台，无多人权限系统；此处验证“资源访问边界”和“操作授权边界”。

- 访问不存在的 `project_id`、`chapter_id`、`scan_id`、`issue_id`、`version_id` 时返回统一错误。
- 不允许用某项目的章节 ID 访问另一个项目的骨架、设定或导出上下文。
- 不允许对不属于该章节当前文本范围的偏移执行段落编辑。
- 不允许通过 QA 修复接口默认整章重写，必须要求受限修改范围。
- 不允许跨章节恢复版本，即某章节不能恢复另一个章节的版本。
- 不允许请求未定义的编辑操作或未定义的导出格式。
- 不允许绕过 `versioning` 直接覆盖历史版本。

## 5. Regression Risks

- 骨架生成后误写入章节正文，破坏模块边界。
- 段落编辑直接覆盖当前正文，导致候选版本与回退能力失效。
- QA 扫描过程中修改正文，造成“诊断即写入”的行为回归。
- 定点修复扩大到整章范围，破坏用户已有内容。
- 版本 diff、快照、回退之间的关联关系不一致，导致历史不可追溯。
- 核心设定更新后未被后续生成/QA 引用，出现上下文漂移。
- 导出内容未反映当前活动版本，而是导出旧版本或候选版本。
- 错误响应未遵循架构规定的统一结构，影响前端处理。

## 6. 测试分层

### 6.1 Unit

关注领域规则、参数校验和边界判断：

- 编辑操作枚举校验。
- 选区偏移合法性校验。
- QA 问题类型、严重度和定位字段组装。
- 版本回退目标归属校验。
- 导出格式校验。
- 错误结构构造。

### 6.2 Integration

关注模块协作、存储和服务编排：

- 项目、设定、骨架、章节之间的数据流转。
- 章节初稿生成对骨架与设定的引用。
- 段落编辑仅产出候选结果，不覆盖正文。
- QA 扫描生成 `QaScan`、`QaIssue` 并保持与 `base_version_id` 关联。
- 定点修复创建新候选版本并限制影响范围。
- 回退后 `current_version_id` 与版本列表的一致性。
- 导出读取当前项目下最新有效内容。

### 6.3 E2E

关注用户可见业务结果：

- 从创建项目到导出内容的完整创作闭环。
- 从 QA 扫描到单条问题定点修复再到版本回退的闭环。
- 从更新角色卡到重新生成/编辑并验证设定生效的闭环。

## 7. 覆盖映射

### 7.1 验收标准覆盖

- AC1、AC10：项目与核心设定管理。
- AC2：骨架生成。
- AC3：章节初稿生成与工作台保存。
- AC4：段落级编辑候选结果。
- AC5、AC7：QA 扫描、角色口吻与设定偏移识别。
- AC6：单条问题定点修复且限制修改范围。
- AC8、AC9：版本比较与回退。
- AC11：`Markdown` / 纯文本导出。
- AC12：单人工作台边界，不覆盖多人协作。

### 7.2 Done 定义覆盖

- API 可正常调用：以集成/E2E 覆盖核心接口。
- 核心功能有测试覆盖：每个能力域至少 1 条 happy path 与 1 条边界/失败路径。
- 使用 `pytest`：测试设计按 `pytest` 结构拆分。
