# Test Plan - FEAT-0001

## 测试目标
- 验证 FEAT-0001 的首发方案可稳定支撑“审稿驱动型章节工坊”主流程。
- 验证人工确认模式与全自动模式均满足产品与架构约束。
- 验证章节生成、评分回流、最佳版本保留、人工复核标记、结构化记忆更新、段落改写等关键链路在正常与异常条件下行为正确。
- 验证权限、配额、内容安全、后台运营接口在受限条件下返回明确错误，不出现 silent fail。

## 范围
- 包含：
  - 小说项目创建、模板选择、总纲与章节大纲维护。
  - 单章生成流程：3 个剧情走向、主编评分选案、正文生成、读者评分、最多 5 次回流、最佳版本保留。
  - 人工确认模式下的方案确认与最终定稿。
  - 全自动模式下单次最多 10 章的串行续写与逐章记忆更新。
  - 段落级重写/扩写、差异对比、一致性检查。
  - 配额校验、权限控制、后台模板审核与运营管理核心接口。
  - 内容安全过滤、版权声明展示、日志留存。
- 不包含：
  - 多人实时协作。
  - 社区能力、配图、多格式导出。
  - 复杂企业审批流与高级 BI 分析。

## 风险点
- 章节编排状态机错误导致跳步、并发推进或重复生成。
- 自动模式超过 10 章限制或未按章节顺序串行执行。
- 回流计数边界错误，导致少回流、超回流或未保留最高分版本。
- 人工模式下错误绕过用户确认直接进入正文生成或最终入库。
- 结构化记忆未及时更新，导致后续章节上下文断裂。
- 段落重写后差异对比缺失或一致性检查失真。
- 权限、配额、内容安全策略被绕过。
- 后台高权限接口越权访问或缺少审计记录。
- Agent/任务基础设施异常时错误码不清晰，影响恢复与排障。

## 测试层次（unit/integration/e2e）
### unit
- 状态机迁移规则：章节状态、任务状态、回流次数上限、10 章上限校验。
- 评分与选优逻辑：主编选最高分方案、读者评分阈值判断、最佳稿件选择。
- 配额规则：免费额度、月度额度、手动调额后的剩余额度计算。
- 权限规则：普通用户、项目拥有者、后台运营角色的访问判定。
- 结构化记忆组装与更新：5 类记忆字段完整性、更新时机、缺失校验。
- 段落改写结果处理：差异对比生成、一致性检查结果归档。

### integration
- FastAPI API 与领域服务、持久化、异步编排之间的集成。
- 单章生成流水线集成：方案生成 -> 主编评分 -> 选案 -> 正文生成 -> 读者评分 -> 回流/完成。
- 自动续写流水线集成：连续 10 章串行、逐章更新记忆、任务状态查询。
- 配额、内容安全、审计日志与业务接口的联动。
- 后台模板审核、套餐配置、调额、订单查询接口与权限体系联动。

### e2e
- 用户从创建项目到完成单章生成、人工确认、定稿入库的完整流程。
- 用户发起全自动续写任务，系统顺序完成多章生成并展示复核标记。
- 用户对正文段落发起重写/扩写并查看差异与一致性提示。
- 运营人员完成模板审核、配额调整、订单查看的后台流程。
- 非法访问、额度不足、上下文缺失、敏感内容命中等失败场景。

## 测试策略
- 单元测试优先覆盖纯规则与边界：
  - 回流上限 `5`
  - 自动续写章节上限 `10`
  - 读者评分通过阈值 `8.0`
  - 最高分版本选择
  - 章节串行约束
- 集成测试通过 mock Agent 响应保证稳定可重复，重点验证：
  - 每章固定产出 3 个方案
  - 主编评分字段完整
  - 自动选最高分方案
  - 低分自动回流
  - 达到回流上限后输出最高分并标记“建议人工复核”
  - 完成一章后再更新记忆并进入下一章
- e2e 测试覆盖真实 API 行为与关键页面流程，关注用户可见结果、错误提示与状态展示。
- 权限与异常测试必须与正常路径同等优先，不允许仅验证 happy path。
- 对 acceptance criteria 建立测试用例映射，确保每条标准至少被一个用例覆盖，关键标准由多个用例正反向覆盖。
- 对关键失败场景校验明确错误码：
  - `400` 参数非法或超限
  - `403` 权限不足
  - `409` 状态冲突
  - `422` 上下文缺失
  - `429` 配额不足
  - `503` Agent 或任务系统异常
- 对日志与审计的验证采用事件断言：
  - 人工确认动作
  - 后台调额动作
  - 章节生成/回流/完成节点
  - 内容安全拦截记录
