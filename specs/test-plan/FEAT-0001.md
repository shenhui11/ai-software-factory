# Test Plan - FEAT-0001

## 测试目标
- 验证“审稿驱动型章节工坊”最终方案满足已确认业务规则，尤其是“3 个走向方案 -> 主编评分筛选 -> 正文生成 -> 读者评分 -> 回流重写 -> 保留最高分版本”的闭环。
- 验证人工确认模式与全自动模式都能在章节续写场景下正确运行，且单次任务最大连续生成 10 章。
- 验证权限、额度、内容安全、任务冲突、Agent 异常等关键风险场景被正确拦截、记录并反馈。
- 验证测试覆盖 Acceptance Criteria，确保后续实现与验收有可追溯映射。

## 范围
- 前台创作主流程：
  - 创建小说项目，配置题材、篇幅、模板、总纲、角色设定、世界观。
  - 根据项目名称与已填写信息自动生成或补全项目基础设定。
  - 发起续写任务，区分人工确认模式与全自动模式。
  - 每章固定生成 3 个剧情走向方案。
  - 主编评分并自动选择最高分方案。
  - 写手生成正文，读者评分，低分回流，最多 5 次。
  - 最终保留最高分版本；5 次后仍未达标时标记“建议人工复核”。
  - 段落级重写、扩写、去除、差异对比与一致性提示。
  - 章节详情页关键操作的处理中、成功、失败反馈与防重复点击。
- 结构化记忆与任务状态：
  - 章节串行推进。
  - 章节状态与任务状态流转。
  - 生成前按需装配上下文，完成后记忆回写与后续章节消费。
- 平台约束与后台能力：
  - 额度校验、任务冲突控制、内容安全拦截。
  - 模板管理、会员额度、订单、手动调额、任务日志。
  - 普通创作者与运营管理员的权限隔离。
- 不在本轮测试重点中的内容：
  - 多用户协同创作。
  - 高并发队列扩展能力。
  - 高级 BI、复杂支付结算、企业级权限细分。

## 风险点
- 章节主流程被跳步执行，例如未生成 3 个方案就进入正文生成，或绕过主编评分。
- 项目基础设定生成覆盖了用户已输入内容，导致用户手工填写的信息被错误改写。
- 题材扩展仅在前端可见，但后端或模板层未兼容，导致建项或生成阶段异常。
- 回流次数、通过阈值、最终版本选择逻辑实现错误，导致未保留最高分版本或无限循环。
- 人工确认模式与全自动模式状态机混淆，出现错误暂停、错误自动推进或重复扣额。
- 单次任务章数上限、运行中任务冲突、额度不足等前置约束未正确拦截。
- 结构化记忆未正确回写，导致后续章节角色、世界观、事件摘要不一致。
- 权限控制缺失，普通创作者可访问后台管理接口，或系统内部接口被外部误用。
- 内容安全、Agent 超时、外部模型异常等失败场景未显式报错或未留审计日志。
- 段落级重写/扩写/去除仅返回文本结果，遗漏差异对比或一致性风险提示。
- 章节详情页按钮缺少处理中反馈或局部禁用控制，导致重复点击、重复请求或用户误判为无响应。
- 结构化记忆虽然入库，但生成前未按相关性裁剪，导致 Prompt 噪声过大或事实遗漏。

## 测试层次（unit/integration/e2e）
### unit
- 状态机与规则函数：
  - 单次最多 10 章校验。
  - 项目基础设定生成时的“保留已有字段、补全缺失字段”规则。
  - 固定 3 个走向方案校验。
  - 主编最高分方案选择逻辑。
  - 读者评分阈值判断。
  - 回流次数上限 5 次控制。
  - 最终最高分版本选择与人工复核标记逻辑。
- 权限与额度规则：
  - 普通用户/管理员鉴权。
  - 配额扣减与不足拦截。
  - 任务冲突检测。
- 章节交互反馈规则：
  - 关键提交动作触发后按钮禁用。
  - 请求结束后显示成功或失败反馈。
- 错误映射：
  - `QUOTA_EXCEEDED`、`TASK_CONFLICT`、`CONTENT_BLOCKED`、`AGENT_TIMEOUT` 等错误码映射。

### integration
- API + 服务层 + 数据层联调：
  - 项目创建与查询。
  - 项目基础设定生成与回填。
  - 续写任务创建、章节串行推进、状态更新。
  - 走向方案、评分结果、正文版本、日志入库。
  - 人工确认模式暂停/确认后继续。
  - 全自动模式连续推进至任务完成或达到上限。
  - 段落重写/扩写/去除的差异结果保存。
  - 章节详情页关键按钮的反馈状态与防重复点击联调。
  - 管理后台模板、额度、订单、安全策略、日志查询联调。
- 异常联调：
  - Agent 超时重试后失败。
  - 内容安全阻断。
  - 运行中重复发起任务。
  - 章节回流耗尽后输出最高分版本。

### e2e
- 以真实业务视角验证：
  - 创作者从建项、AI 生成基础设定到人工确认完成单章生成。
  - 创作者发起全自动多章续写并查看章节进度、评分、复核标记。
  - 创作者对已生成段落执行重写/扩写/去除并查看 diff。
  - 管理员审核模板、调整额度、查看日志。
  - 非法角色访问后台、额度不足、敏感内容命中、任务冲突等完整用户路径。

## 测试策略
- 采用风险优先策略，先覆盖章节主闭环、状态机、权限、异常处理，再补充后台与边界条件。
- Acceptance Criteria 全量建立映射，确保每条验收标准至少被 1 个测试用例覆盖，关键规则由多个用例交叉验证。
- 单元测试优先固化确定性业务规则；集成测试验证 API、服务、持久化和日志链路；E2E 验证真实用户路径与页面/接口联动。
- 对外部 Agent 调用采用可控桩数据，重点验证编排顺序、重试、状态流转、错误分类与最终落库结果，不依赖模型随机输出。
- 对项目基础设定生成验证“空字段补全、部分字段保留、未知题材降级可用”三类场景。
- 对结构化记忆验证采用“生成前装配内容 + 生成后回写结果”的双向校验，避免只测落库不测消费。
- 对评分相关流程使用边界值设计：
  - `7.9` 触发回流。
  - `8.0` 直接通过。
  - 第 5 次回流后仍 `< 8.0` 时停止并标记人工复核。
- 对章数、方案数、权限、额度等采用等价类与边界值结合：
  - 章数 `1`、`10`、`11`。
  - 方案数固定 `3`，拒绝 `2` 或 `4`。
  - 权限区分创作者、管理员、系统任务执行器。
- 对可观测性进行验证，确保关键路径均写入任务日志、错误日志与审计记录，避免 silent fail。
- 对前端反馈进行验证，确保所有关键异步动作至少有一个可见进行中状态，且处理完成后界面能给出结果反馈。
