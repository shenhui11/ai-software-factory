# Test Plan - FEAT-0001

## 测试目标
- 验证首期“审稿驱动型章节工坊”满足 PRD、架构和最终决策中的核心业务流程与约束。
- 验证人工确认模式与全自动续写模式均可稳定运行，且不突破章节数、回流次数、评分阈值等硬规则。
- 验证权限、配额、内容安全、任务状态和审计行为在正常与异常场景下均返回明确结果，不出现 silent fail。
- 验证测试用例对 acceptance criteria 建立完整映射，覆盖核心功能路径、权限路径、异常路径和边界条件。

## 范围
- 覆盖范围：
  - 小说项目创建与模板选择。
  - 总纲、章节大纲、章节正文三层结构维护。
  - 单章生成流水线：3 个走向候选、主编评分选优、正文生成、读者评分、自动回流、最高分版本保留。
  - 人工确认模式：查看评分说明、确认走向、最终定稿。
  - 全自动续写模式：单次最多 10 章、逐章串行、任务状态展示。
  - 段落级重写与扩写、差异对比、上下文一致性检查。
  - 结构化记忆：全书摘要、角色卡、世界观设定、已发生事件摘要、章节状态。
  - 后台能力：模板审核、会员套餐、免费额度/月度额度调整、订单管理、手动调额、基础运营报表入口。
  - 内容安全、版权声明、日志与数据留存。
- 不覆盖范围：
  - 多用户实时协作。
  - 社区互动、AI 配图、多格式导出。
  - 复杂企业审批流和高级 BI。

## 风险点
- 章节状态机或任务状态机迁移错误，导致重复生成、跳步或并行推进。
- 自动续写未限制在 10 章内，或未按章节顺序串行执行，造成记忆错位。
- 走向候选不是固定 3 个，或主编评分缺字段、选优逻辑错误。
- 读者评分低于 8.0 时未触发回流，或回流次数超过 5 次。
- 达到上限后未保留最高分版本，或未标记“建议人工复核”。
- 人工确认模式被绕过，系统直接沿用自动选中的方案或直接入库。
- 结构化记忆缺项、更新时机错误，导致后续章节上下文不一致。
- 段落级编辑未返回 diff 或一致性检查结果，影响人工审稿。
- 普通用户越权访问后台接口，或配额不足、敏感题材命中时缺少清晰错误码。
- Agent/任务系统异常时接口返回不稳定，影响重试与排障。

## 测试层次（unit/integration/e2e）
### unit
- 章节与任务状态机迁移规则。
- `target_chapter_count <= 10` 校验。
- `reader_score >= 8.0` 通过规则。
- 最大回流次数 `5` 校验。
- 主编最高分选优与正文最高分版本保留规则。
- 权限判定、配额计算、内容安全命中判定。
- 结构化记忆拼装与章节完成后的更新逻辑。

### integration
- FastAPI API、应用服务、持久化和异步编排的联动。
- 单章流水线：走向生成 -> 主编评分 -> 选优 -> 正文生成 -> 读者评分 -> 回流/完成。
- 自动续写任务：连续多章串行、逐章更新记忆、状态查询。
- 段落级编辑与 diff/一致性检查结果落库。
- 配额、内容安全、审计日志与业务接口的联动。
- 后台模板审核、调额、订单查询与权限控制。

### e2e
- 用户创建项目并完成单章人工确认生成。
- 用户发起自动续写任务并查看最多 10 章的任务结果。
- 用户对正文段落执行 rewrite/expand 并查看差异与一致性提示。
- 运营人员执行模板审核与额度调整。
- 用户或系统遭遇权限不足、参数超限、前置条件缺失、额度不足、内容安全拦截、Agent 失败等场景。

## 测试策略
- 单元测试优先覆盖硬规则和边界值：
  - 候选方案数固定为 `3`
  - 自动续写上限为 `10`
  - 评分阈值为 `8.0`
  - 回流上限为 `5`
  - 完成后保留最高分版本
- 集成测试使用可控的 Agent mock/stub，验证编排顺序、字段完整性、错误码和持久化结果。
- e2e 测试覆盖用户可见主流程与关键失败路径，确保不是只验证 happy path。
- 权限测试必须同时覆盖普通用户、项目拥有者、后台运营角色三类身份。
- 异常测试必须覆盖并断言明确错误码：
  - `400` 参数非法或超限
  - `403` 权限不足
  - `404` 资源不存在
  - `409` 状态冲突
  - `422` 前置条件缺失
  - `429` 配额不足
  - `503` Agent 或任务系统不可用
- 审计验证采用事件断言，至少覆盖：
  - 任务创建
  - 走向评分与选优
  - 回流次数增加
  - 人工确认/定稿
  - 模板审核
  - 手动调额
  - 内容安全拦截
- 测试用例文档必须将 acceptance criteria 映射到具体案例；关键标准至少要有一个正向用例和一个反向或边界用例覆盖。
