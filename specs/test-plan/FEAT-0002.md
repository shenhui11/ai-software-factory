# FEAT-0002 Test Plan

## 1. 测试策略

### 1.1 测试目标

验证 `FEAT-0002` 的 MVP 是否满足“成长积分型会员体系”的最小可用闭环：

- 登录用户具备明确会员状态、等级、成长值和积分账户。
- 用户可通过签到与任务完成获得积分，并可查看面板、任务、积分流水和奖励记录。
- 系统可基于成长门槛自动升级会员等级，并即时展示已解锁权益。
- 运营人员可通过后台配置任务、等级门槛和权益映射，并通过发布使配置生效。
- 系统对重复签到、重复任务完成和异常高频发分具备最小必要限制，并保留日志与审计记录。

### 1.2 测试范围

覆盖 PRD 验收标准、Architecture API 契约、权限边界和一致性约束：

- 用户侧会员档案、面板、任务、签到、任务完成、积分流水、奖励记录。
- 运营侧会员配置读取、更新、发布和指定用户会员详情查询。
- 积分记账、等级评估、权益映射、奖励发放、配置版本生效。
- 幂等控制、频率限制、统一错误结构、日志和审计事件。

### 1.3 不测试项

- 支付、订阅、自动续费、退款与发票。
- 积分商城、多商品兑换和库存能力。
- 企业组织、席位和 SSO。
- 复杂规则引擎、设备指纹和完整反作弊平台。
- 多 SKU 权益包、复杂 CRM 和高级运营分析看板。

### 1.4 方法与原则

- 以 FastAPI 接口行为验证为主，保证 API 能正常调用并符合统一错误结构。
- 以状态变化和审计事实断言为主，不只断言响应成功，还要断言会员档案、积分账户、流水、奖励、配置版本的一致性。
- 重点验证架构边界：
  - `risk_control` 必须先于记账执行。
  - 积分余额变化必须伴随 `PointsLedgerEntry`。
  - 等级升级必须通过 `level_engine` 统一计算。
  - 用户侧业务参数必须来源于已发布 `ops_config`，不得依赖硬编码。
- 优先覆盖主链路闭环，再覆盖高风险边界、越权访问和发布配置后的回归风险。

## 2. Happy Path

### 2.1 用户成长闭环

1. 用户登录后查询 `GET /api/v1/members/me`，看到会员状态、当前等级、成长值、下一等级和积分余额。
2. 用户查询 `GET /api/v1/members/me/dashboard`，看到成长进度、已解锁权益、待完成任务和最近奖励。
3. 用户查询 `GET /api/v1/members/me/tasks`，看到当前配置下启用的任务及奖励积分。
4. 用户执行 `POST /api/v1/members/me/check-ins` 完成首次签到，系统发放积分并写入流水。
5. 用户执行 `POST /api/v1/members/me/tasks/{task_code}/complete` 完成一个有效任务，系统继续发分并记录来源。
6. 用户累计成长值达到等级门槛后，系统自动升级会员等级并解锁对应权益。
7. 用户再次查看会员档案、面板和奖励记录，确认等级、权益和奖励结果已生效。
8. 用户查询积分流水，能追溯每次积分增加的来源、时间和余额变化。

### 2.2 运营配置闭环

1. 运营人员读取当前配置快照。
2. 运营人员分别更新任务配置、等级门槛和权益映射。
3. 运营人员发布新配置版本。
4. 用户侧后续查询任务、签到或触发升级时，读取并应用最新发布版本。

## 3. Edge Cases

- 未登录访问用户侧接口，返回鉴权失败和统一错误结构。
- 用户存在会员档案但积分账户缺失时，系统应显式失败或完成初始化，不能返回部分成功的脏状态。
- 同一自然日重复签到，第二次请求不得重复加分或重复写流水。
- 相同 `request_id` 重复提交任务完成，只能记账一次。
- 任务已禁用、任务不存在或未发布时，任务完成请求必须被拒绝。
- 任务达到 `daily_limit` 后再次触发，不得继续发分。
- 用户完成任务后积分增加但未达到门槛时，不得误升级。
- 升级触发后若奖励发放失败，接口必须显式报错并具备可追溯审计信息。
- 积分流水查询传入非法 `change_type`、无效 `cursor` 或超限 `limit` 时，返回统一错误结构。
- 用户无任务、无奖励或无流水时，查询接口应返回合法空集合而非 5xx。
- 配置已更新但尚未发布时，用户侧不应读取到未发布配置。
- 等级门槛顺序错误、等级编码重复、权益映射引用无效等级或无效权益时，发布必须失败。
- 高频重复提交触发风控时，应返回可识别的拦截结果和 `request_id`。
- 查询不存在的 `user_id`、`task_code` 或配置版本时，应返回统一错误结构。

## 4. Permission Cases

- 普通登录用户只能访问自己的 `/api/v1/members/me*` 资源。
- 普通用户不得访问任意 `/api/v1/admin/*` 接口。
- 普通用户不得通过篡改路径参数、查询参数或请求体读取他人会员档案、积分流水和奖励记录。
- 运营人员可管理配置并查询指定用户会员详情，但权限范围仅限配置与排查，不应获得未定义的人工发分或回滚能力。
- 运营配置更新和发布必须校验后台角色，普通登录态或只读身份不能绕过。
- 指定用户查询接口必须限制为单用户排查，不允许无边界批量导出用户数据。
- 任意关键变更请求都应记录操作者身份、请求号和配置版本，避免匿名操作。

## 5. Regression Risks

- 签到或任务完成绕过幂等校验，导致重复记账和刷分。
- `PointsAccount` 与 `PointsLedgerEntry` 不一致，出现余额变化但无流水或流水存在但余额未更新。
- 升级和奖励发放不在同一业务事务或请求上下文中，导致状态部分成功。
- 用户面板未读取最新等级或权益映射，升级后展示仍是旧权益。
- 配置发布后用户侧仍读取旧版本，导致任务积分值、门槛或权益映射与后台不一致。
- 风控拦截被静默吞掉，前端无法区分“未完成任务”和“请求被拒绝”。
- 越权访问导致用户可读取或推断他人会员数据。
- 运营配置发布缺少审计事件，后续无法追溯异常等级变化或积分异常。
- 错误响应不符合架构约定，影响前端处理和排障链路。

## 6. Unit / Integration / E2E 分类

### 6.1 Unit

关注领域规则和纯业务判断：

- `membership_status` 与 `current_level` 计算。
- 下一等级差值计算。
- 签到自然日去重判断。
- 任务重复完成和 `daily_limit` 校验。
- 积分余额变更与累计值计算。
- 等级门槛排序、重复编码和配置有效性校验。
- 权益映射解析和启停判断。
- 风控拦截条件与错误码构造。

### 6.2 Integration

关注应用服务、存储和模块协作：

- `CheckInService` 串联风控、记账、升级评估、奖励发放和审计事件。
- `TaskCompletionService` 与任务记录、积分流水、会员档案的一致性。
- `MemberDashboardService` 聚合会员档案、任务、权益、奖励和积分信息。
- `MemberConfigService` 更新与发布配置版本，并被用户侧读取。
- 用户侧和运营侧接口统一错误结构、鉴权与角色校验。

### 6.3 E2E

关注用户和运营人员实际可见结果：

- 从登录查询会员档案到签到、升级、查看权益和流水的完整成长闭环。
- 从完成任务到发分、升级、发奖、查询奖励记录的完整业务闭环。
- 从运营读取配置到更新、发布，再由用户侧读取并生效的完整配置闭环。

## 7. 覆盖映射

### 7.1 验收标准覆盖

- AC1：会员状态与等级展示。
- AC2：积分账户与积分流水查询。
- AC3：任务完成自动发分与来源记录。
- AC4：达到门槛自动升级。
- AC5：会员面板展示成长进度、权益和待完成任务。
- AC6：等级权益映射真实生效。
- AC7：运营配置无需改代码即可生效。
- AC8：重复签到和异常高频获取积分受限。
- AC9：升级、发分、发奖具备事件记录和日志。
- AC10：实现遵循 FastAPI、pytest 和无敏感硬编码约束。

### 7.2 Done 定义覆盖

- API 可正常调用：通过 Integration 和 E2E 覆盖核心用户侧与运营侧接口。
- 每个核心功能有测试覆盖：签到、任务、积分、升级、权益、配置、权限、风控均有覆盖。
- 使用 `pytest`：测试设计按 `pytest` 的 unit / integration / e2e 分层组织。
