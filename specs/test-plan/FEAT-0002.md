# FEAT-0002 Test Plan

## 1. 测试策略

### 1.1 测试目标

验证 `FEAT-0002` 的 MVP 是否满足“成长积分型会员体系”的最小闭环：

- 登录用户具备明确会员状态、等级和积分账户。
- 用户可通过签到和任务完成获得积分，并可查看会员面板、任务、积分流水和奖励记录。
- 系统可基于积分或成长值自动升级，并即时生效等级权益。
- 运营人员可配置任务、等级门槛和权益映射，并通过发布使配置生效。
- 系统对重复签到、重复任务触发和异常高频获取积分具备最小必要限制，并保留日志与审计记录。

### 1.2 测试范围

覆盖 PRD 验收标准、Architecture API 设计、权限模型和安全约束：

- 用户会员档案与会员面板查询。
- 任务中心与每日签到。
- 手动提交任务完成与积分发放。
- 积分账户与积分流水查询。
- 等级评估、自动升级和权益映射。
- 奖励记录查询与基础奖励发放。
- 运营侧配置读取、更新、发布。
- 指定用户会员详情查询。
- 幂等控制、频率限制、基础反重放、审计与错误结构。

### 1.3 不测试项

- 支付、订阅、自动续费、退款和发票。
- 积分商城、多商品兑换和库存管理。
- 企业组织、席位、SSO。
- 完整反作弊平台、设备指纹和复杂风控模型。
- 多 SKU 权益商品化和复杂规则引擎。

### 1.4 方法与原则

- 以 FastAPI API 行为验证为主，确保接口契约、错误结构和权限边界符合架构文档。
- 以状态变化和审计事实断言为主，重点验证 `PointsLedgerEntry`、会员等级、奖励记录、配置版本的关联一致性。
- 优先验证主链路闭环：任务完成 -> 积分发放 -> 流水记账 -> 等级评估 -> 权益生效 -> 可查询可追溯。
- 对高风险场景强化验证：
  - 同一自然日签到幂等。
  - 同一任务请求不能重复记账。
  - 风控拒绝必须显式返回，不允许 silent fail。
  - 任意余额变化都必须存在对应流水。
  - 配置必须来自 `ops_config`，不依赖代码硬编码。

## 2. Happy Path

### 2.1 用户成长闭环

1. 用户登录后查询自己的会员档案，看到会员状态、当前等级、成长值、下一等级和积分余额。
2. 用户进入会员面板，看到成长进度、已解锁权益、待完成任务和最近奖励。
3. 用户读取任务列表，看到启用中的签到任务和其他成长任务。
4. 用户执行当日签到，系统返回签到成功、发放积分、当前等级和是否升级。
5. 用户完成一个首版支持的任务，系统自动发放积分并记录流水。
6. 用户积分或成长值达到门槛后，系统自动升级会员等级。
7. 用户重新查看会员面板，看到等级变化、权益变化和奖励结果。
8. 用户查询积分流水和奖励记录，可理解积分与会员状态变化来源。

### 2.2 运营配置闭环

1. 运营人员读取当前会员配置快照。
2. 运营人员更新任务配置、等级门槛和权益映射。
3. 运营人员发布配置版本。
4. 后续用户查询任务、执行签到或触发升级时，读取到最新配置并生效。

## 3. Edge Cases

- 未登录访问用户侧接口，返回鉴权失败。
- 会员档案已存在但积分账户不存在，系统应返回明确错误或完成初始化，不能返回不一致数据。
- 用户当日重复签到，第二次请求不得重复加分。
- 相同 `request_id` 重复提交任务完成，请求可幂等返回，但不得重复生成积分流水。
- 禁用任务仍被客户端提交完成请求，系统应拒绝。
- 任务达到每日上限后再次触发，系统应拒绝继续发分。
- 积分发放成功但升级门槛未达成时，不得误升级。
- 升级成功但奖励发放失败时，接口需显式返回错误并保留可追溯记录。
- 积分流水查询使用非法 `change_type`、超限 `limit` 或无效 `cursor` 时返回统一错误结构。
- 用户无奖励记录、无流水记录或无可见任务时，查询接口仍需返回合法空集合。
- 配置更新未发布前，不应影响用户侧运行结果。
- 配置存在无效等级门槛顺序、重复等级编码或无效权益映射时，发布应失败。
- 高频重复请求触发基础风控时，应返回拒绝结果和 `request_id`。
- 查询不存在的 `user_id`、`task_code` 或配置版本时，应返回统一错误结构。

## 4. Permission Cases

- 普通登录用户只能访问自己的 `/members/me`、`/dashboard`、`/tasks`、`/points/ledger`、`/rewards`。
- 普通用户不得访问任意 `/api/v1/admin/*` 接口。
- 运营人员可读取和发布会员配置、查询指定用户会员详情，但不得执行平台管理员专属的人工修正、回滚、补发操作。
- 若存在目标 `user_id` 参数或内部资源引用，必须校验当前用户与目标用户一致，禁止越权读取他人会员信息。
- 运营配置接口必须校验角色；缺少权限时不能通过“只读 token”或普通登录态绕过。
- 任何人工调整相关接口若在首版暴露，必须校验管理员角色、操作者身份、原因和请求号；否则接口不应可用给普通用户或运营角色。

## 5. Regression Risks

- 签到或任务完成绕过幂等控制，导致重复记账和刷分。
- 积分余额变化未同步生成流水，导致账户与审计事实不一致。
- 等级升级与奖励发放不在同一业务上下文内，出现“已升级但未发奖”或“已发奖未升级”。
- 配置发布后用户侧仍读取旧版本，导致任务积分值、等级门槛或权益映射不一致。
- 权益映射生效失败，用户升级后在面板上看不到已解锁权益。
- 风控拒绝改为静默忽略，前端无法区分“未达成”和“被拒绝”。
- 用户接口错误暴露他人账户信息，出现越权读取。
- 运营接口缺少角色审计，无法追踪配置变更和异常发分来源。
- 错误响应不符合统一结构，导致前端和排障链路不稳定。

## 6. Unit / Integration / E2E 分类

### 6.1 Unit

关注领域规则和纯业务判断：

- 会员状态与等级计算规则。
- 下一等级差值计算。
- 每日签到自然日幂等判断。
- 任务每日上限和重复完成校验。
- 积分记账前后余额计算。
- 配置版本选择与门槛排序校验。
- 权益映射解析。
- 风控拒绝条件和错误码构造。

### 6.2 Integration

关注应用服务、持久化与模块协作：

- `CheckInService` 调用风控、记账、升级评估、奖励发放的完整协作。
- `TaskCompletionService` 与 `points_ledger`、`member_profile`、`reward_dispatch` 的事务一致性。
- `MemberDashboardService` 聚合会员档案、权益、任务和奖励结果。
- `MemberConfigService` 更新和发布配置版本后，被用户侧服务正确读取。
- 用户与运营接口统一错误结构、审计事件和结构化日志写入。

### 6.3 E2E

关注用户和运营人员可见结果：

- 从登录查询会员档案到签到、升级、查看权益和流水的完整用户成长闭环。
- 从完成任务到发分、升级、发奖、查询奖励记录的完整业务闭环。
- 从读取配置到更新任务/等级/权益并发布，再由用户侧读取新配置生效的运营闭环。

## 7. 覆盖映射

### 7.1 验收标准覆盖

- AC1：会员状态与等级展示。
- AC2：积分账户与流水查询。
- AC3：任务完成自动发分及来源记录。
- AC4：达到门槛自动升级。
- AC5：会员面板展示成长进度、权益和任务。
- AC6：等级权益映射真实生效。
- AC7：运营配置无需改代码即可生效。
- AC8：重复签到与异常高频获取积分受限。
- AC9：升级、发分、发奖有事件记录和日志。
- AC10：实现符合 FastAPI、pytest 与无敏感硬编码约束。

### 7.2 Done 定义覆盖

- API 可正常调用：通过集成和 E2E 覆盖核心用户侧与运营侧接口。
- 每个核心功能有测试覆盖：签到、任务、积分、升级、权益、配置、权限、风控均有主路径和边界路径。
- 使用 `pytest`：测试分层按 `pytest` 常见组织方式设计。
