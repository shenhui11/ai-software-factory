# FEAT-0003 Test Plan

## 1. 测试策略

### 1.1 测试目标

验证 `FEAT-0003` 的 MVP 是否满足“模板资源库平台”的最小闭环：

- 系统能够清晰区分 `系统模板` 与 `用户模板`，并保证基础字段、状态与归属信息一致。
- 管理员能够在后台完成系统模板、分类和标签的创建、编辑、发布、下线、删除与筛选管理。
- 普通用户能够浏览系统模板、管理自己的用户模板、复制系统模板为个人模板，并将模板用于内容或任务创建入口。
- AI 生成功能能够基于一句话需求返回可编辑模板草稿，并允许用户保存为个人模板。
- 权限边界、输入校验、统一错误结构和审计要求符合架构文档与 Definition of Done。

### 1.2 测试范围

覆盖 PRD 验收标准、Architecture API 设计、权限模型和安全约束：

- 前台模板列表、模板详情与模板使用入口。
- 用户模板创建、编辑、删除、查看。
- 系统模板后台创建、编辑、发布、下线、删除与筛选。
- 分类与标签管理。
- 系统模板复制为用户模板。
- AI 模板草稿生成与保存。
- 模板类型区分、状态流转、事务一致性、错误结构和审计记录。

### 1.3 不测试项

- 模板社区、评分、评论、收藏与公开市场。
- 付费模板、会员权益和商业化结算。
- 多轮 AI 对话、多方案候选和复杂编排。
- 团队共享空间、企业级多层权限和复杂协作。
- 模板版本回滚、审核流、灰度发布和复杂治理工作流。

### 1.4 方法与原则

- 以 FastAPI API 行为验证为主，确保接口契约、状态流转、错误结构和权限判断符合架构定义。
- 以资源归属和可见性断言为核心，重点验证 `template_type`、`status`、`owner_user_id`、`source_template_id` 的一致性。
- 优先验证主链路闭环：后台发布系统模板 -> 前台可见 -> 用户查看详情 -> 复制为个人模板 -> 编辑保存 -> 作为创建起点使用。
- 对高风险场景强化验证：
  - 普通用户不能修改系统模板原文。
  - 非拥有者不能查看或修改他人用户模板。
  - 未发布系统模板不得暴露给普通用户。
  - 模板复制必须固化快照，源模板后续修改不得影响已复制用户模板。
  - AI 生成成功不等于模板已入库，保存动作需显式触发。

## 2. Happy Path

### 2.1 管理后台模板治理闭环

1. 管理员进入后台查询系统模板列表，按状态、分类和关键词筛选模板。
2. 管理员创建模板分类与标签，并创建一条系统模板草稿。
3. 管理员编辑系统模板的名称、描述、正文、分类、标签和状态。
4. 管理员发布系统模板，前台用户侧列表可见该模板。
5. 管理员可在后续将模板下线或删除，并得到明确结果。

### 2.2 用户模板复用闭环

1. 普通用户进入模板列表，看到已发布系统模板和自己的用户模板。
2. 用户查看某个系统模板详情，确认内容后执行复制。
3. 系统生成新的用户模板，并记录来源模板关系。
4. 用户编辑复制后的个人模板并保存。
5. 用户在模板详情或列表中执行“使用模板”，获得标准化起始载荷。

### 2.3 AI 草稿闭环

1. 普通用户输入一句话需求，请求 AI 生成模板草稿。
2. 系统返回可编辑的草稿字段，包括名称、描述、内容和建议分类/标签。
3. 用户编辑草稿后保存为个人模板。
4. 用户在“我的模板”中可以查询到新保存的模板。

## 3. Edge Cases

- 创建或发布模板时缺失 `name`、`content`、有效 `category_id` 等必填字段，应返回统一错误结构。
- 普通用户请求模板列表时，不应看到系统草稿或已下线模板。
- 查询不存在的 `template_id`、`category_id`、`tag_id` 时，应返回明确错误。
- 用户模板列表为空、标签为空或分类为空时，查询接口应返回合法空集合。
- 复制系统模板后，源系统模板被编辑或下线，已复制的用户模板内容不应被回写覆盖。
- 用户创建或编辑模板时提交超长文本、非法标签或无效分类，应被拒绝。
- 管理员发布已满足条件的草稿模板成功；发布缺少必填字段的草稿模板失败。
- 删除系统模板时，不应级联删除基于其复制生成的用户模板。
- AI 生成接口超时、模型失败或返回非结构化结果时，应返回明确错误，不得生成脏模板记录。
- 用户对不存在或无权限访问的模板执行 `:use`、`:clone` 或详情查询时，应返回统一错误结构。

## 4. Permission Cases

- 未登录用户访问模板列表、详情、AI 草稿生成或用户模板接口时，应返回鉴权失败。
- 普通登录用户只能查看已发布系统模板和自己的用户模板。
- 普通用户不得访问任意 `/api/v1/admin/*` 模板、分类、标签管理接口。
- 普通用户不得直接编辑、删除或发布系统模板，即使知道 `template_id` 也不能绕过。
- 普通用户不得查看、编辑、删除其他用户的用户模板。
- 管理员可查看后台系统模板的草稿、已发布和已下线状态，并执行状态流转和删除操作。
- 管理员的后台分类和标签维护能力不得自动赋予其读取任意用户私有模板内容的权限，除非架构明确开放；首版按文档仅保证后台系统模板治理。

## 5. Regression Risks

- 模板类型判断错误，导致系统模板与用户模板混淆，出现错误可见性或错误写入路径。
- 系统模板发布/下线状态未正确影响前台可见范围，导致未发布内容泄漏或已发布内容不可见。
- 模板复制未固化快照，导致用户模板被源模板后续修改污染。
- 用户模板所有权校验失效，导致越权查看、编辑或删除他人模板。
- 分类/标签更新与模板主体更新不在同一事务内，导致模板与标签关系部分写入。
- AI 草稿生成结果被错误自动持久化，导致“生成即入库”的行为偏离架构设计。
- 模板删除错误级联影响用户模板，破坏个人资产稳定性。
- 错误响应不符合统一结构，前端无法稳定处理失败场景。
- 后台操作缺少审计事件，无法追踪系统模板治理变更。

## 6. Unit / Integration / E2E 分类

### 6.1 Unit

关注领域规则和纯业务判断：

- `template_type` 与 `status` 的合法取值与转换规则。
- 系统模板发布前必填校验。
- 用户模板所有权判断。
- 系统模板可见性判断。
- 模板复制时的字段映射与 `source_template_id` 赋值。
- 分类、标签合法性与空值校验。
- AI 草稿结果字段校验与错误码构造。

### 6.2 Integration

关注应用服务、持久化与模块协作：

- `TemplateQueryService` 聚合系统模板与当前用户模板的查询行为。
- `SystemTemplateAdminService` 与分类、标签、审计、状态流转的协作。
- `UserTemplateService` 的创建、编辑、删除与资源归属校验。
- `TemplateCloneService` 与模板查询、写库、来源关系持久化的一致性。
- `TemplateGenerationService` 与 `AiGateway`、生成记录、保存动作的解耦关系。
- 各接口统一错误结构、鉴权失败和 404/403 边界响应。

### 6.3 E2E

关注管理员与普通用户可见结果：

- 后台创建并发布系统模板后，前台用户可见并可查看详情的完整闭环。
- 用户复制系统模板、编辑个人模板、查询“我的模板”、执行模板使用的完整闭环。
- 用户一句话生成 AI 草稿、编辑并保存为个人模板的完整闭环。
- 管理员维护分类标签并基于状态、分类、关键词筛选系统模板的完整闭环。

## 7. 覆盖映射

### 7.1 验收标准覆盖

- AC1：双模板体系与数据层清晰区分。
- AC2：模板基础字段与必填校验。
- AC3：管理员可创建、编辑、删除、发布、下线系统模板。
- AC4：管理员可维护分类、标签并筛选系统模板。
- AC5：普通用户可创建、编辑、删除和查看自己的用户模板。
- AC6：普通用户不能直接修改系统模板，但可复制为个人模板。
- AC7：非拥有者不能编辑其他用户模板。
- AC8：前台提供模板列表并区分系统模板和我的模板，可查看详情。
- AC9：用户可使用模板作为创建内容或任务的起点。
- AC10：一句话 AI 生成功能返回可编辑草稿并允许保存。
- AC11：AI 功能只验证单轮生成。
- AC12：实现符合 FastAPI、pytest 与无敏感硬编码约束。

### 7.2 Done 定义覆盖

- API 可正常调用：通过集成和 E2E 覆盖前台与后台核心接口。
- 每个核心功能有测试覆盖：模板查询、治理、复制、AI 生成、权限、状态流转均有主路径与边界路径。
- 使用 `pytest`：测试分层按 `pytest` 组织方式设计。
